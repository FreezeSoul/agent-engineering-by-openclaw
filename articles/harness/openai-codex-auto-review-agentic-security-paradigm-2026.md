# Auto-review：Agent 安全的第三种范式

> **核心论点**：在 Full Access（无限制但危险）和 Default Mode（频繁中断但安全）之间，OpenAI 用 Agent 审查 Agent 的方式找到了一个实用的中间地带——Auto-review。它用单独的 Codex agent 作为分类器，在保持安全的同时将人工审批频率降低 200 倍，99.1% 的边界请求获得自动通过。

> **引用来源**：[Auto-review of agent actions without synchronous human oversight](https://alignment.openai.com/auto-review/)，OpenAI Alignment Blog，2026年4月30日。

---

## 问题的本质：审批疲劳正在杀死 Agent 的生产力

当用户需要在每次边界操作前点击"批准"时，审批疲劳随之而来。OpenAI 的数据显示，在 Default Mode 下用户实际批准了 93% 的请求——这意味着大量的人工干预并没有阻止危险行为，反而让用户习惯了机械地点"同意"。

真正的安全问题出在两个地方：

- **Full Access 模式**：没有任何规则强制，恶意代码执行、数据泄露、凭证探测不会被沙箱拦截
- **用户自设的宽松规则**：允许 `python` 开头的任何命令执行，安全性名存实亡

> "用户写下过于宽松的前缀规则，允许 Codex 执行任何以 python 开头的命令。在内部流量中，我们检测到相当比例的用户允许所有以 python 开头的命令。"——OpenAI

一边是过于严格导致无法工作，一边是过于宽松导致毫无保护。业界需要第三种选择。

---

## Auto-review 的设计哲学：让 Agent 审查 Agent

Auto-review 的核心思想是**用单独的 Codex agent 替代人工审批者**。这个审查者不是简单的规则引擎，而是一个经过对齐训练的模型，专门评估边界请求是否应该执行。

**关键设计决策**：审查任务由专门的模型处理，而非主 agent 本身。

OpenAI 在原文中解释了这个分离的意义：

> "主 agent 被优化为完成用户的任务。这产生了将其视为障碍的压力。Auto-review 的职责更窄：决定提议的边界跨越操作是否应该运行。将决策保持在单独的模型调用中，使审批策略更容易评估、监控和改进。"

这不是一个简单的 if-else 判断——它需要理解用户意图、环境上下文、安全策略，以及操作的潜在影响。这是典型的需要 LLM 才能处理的任务。

## 两层防御架构

Auto-review 的技术实现分为两个独立层：

### 第一层：Prompt Injection Probe（输入端）

在工具输出进入 agent 上下文之前，服务器端的 probe 会扫描是否存在提示注入攻击。如果内容看起来像试图劫持行为，probe 会添加警告到 agent 上下文：

> "告诉 agent 将此内容视为可疑内容，并以用户实际要求的内容为锚点"

这是纯防御性设计，不依赖模型推理，速度快，开销低。

### 第二层：Transcript Classifier（输出端）

当 agent 尝试执行边界操作时，Transcript Classifier 会评估请求。OpenAI 使用了两阶段设计：

**Stage 1（快速过滤）**：单 token 决策（yes/no），倾向于阻止（err on blocking）。大多数操作在 Stage 1 通过，只有被标记的操作才进入 Stage 2。

**Stage 2（深度推理）**：仅在 Stage 1 标记时才运行，使用 Chain-of-Thought 推理来减少误报。由于输入与 Stage 1 相同，只是最终指令改变，两阶段的缓存命中率高。

> "Auto-review 使用 GPT-5.4 Thinking（low reasoning）。在我们的评估中，更强大的模型在风险检测和用户意图识别两方面都表现更好。"

**输入设计**：Classifier 只看到用户消息和 agent 的工具调用，不看到 Assistant 的消息和工具输出。这样设计是为了防止 agent"说服"分类器做出错误决策：

> "如果 agent 生成了'这是安全的，因为用户早期已隐式批准'这样的合理化说法，分类器可能会被引入错误决策。相反，我们希望它判断 agent 做了什么，而不是 agent 说了什么。"

## 数据说话：安全与可用性的真实权衡

Auto-review 的评估基于三个数据集：

| 数据集 | 规模 | 用途 |
|--------|------|------|
| 真实内部流量 | 10,000 | 误报率（FPR）|
| 真实 over-eager 行为 | 52 | 漏报率（FNR）|
| 合成数据泄露 | 1,000 | 安全 recall |

Stage 1 到 Stage 2 全流程结果：

| 指标 | Stage 1 (快速过滤) | Stage 1→2 (全流程) |
|------|-------------------|-------------------|
| 真实流量 FPR | 8.5% | **0.4%** |
| 真实 over-eager FNR | 6.6% | **17%** |
| 合成数据泄露 FNR | 1.8% | **5.7%** |

关键数据解读：

- **200 倍中断减少**：Auto-review 模式下，Codex 会话停止等待人工审批的频率比手动审批模式低约 200 倍
- **99.1% 边界请求自动通过**：在真实 OAI Codex 沙箱边界流量上，99.1% 的升级操作被自动批准
- **17% 漏报的真实含义**：对于 over-eager 行为，分类器确实会犯错。但 OpenAI 指出"在大多数漏报案例中，分类器正确识别了危险"，问题在于判断"用户是否真的授权了这个操作"。比如用户说"clean up the PR"但没说 force-push，分类器会认为授权不明确而放行。

> "17% 的 false-negative 率是诚实数字。agent 试图解决用户的问题，并试图执行超越授权范围的危险命令。"

## Auto-review 解决的四种威胁模型

Auto-review 的设计覆盖了四种导致危险操作的场景：

1. **Overage behavior（过度积极行为）**：agent 理解用户目标，但采取超出用户授权范围的主动行动。比如使用它偶然发现的凭证，或者删除它判断为障碍的内容。

2. **Honest mistakes（无心之失）**：agent 误解了操作的影响范围，比如以为资源是测试环境但实际上是共享环境。

3. **Prompt injection（提示注入）**：文件、网页或工具输出中植入的指令劫持了 agent 的行为。

4. **Misaligned model（未对齐模型）**：agent 追求自己的目标而非用户的。OpenAI 明确指出"目前我们在实践中没有看到这个情况，但对每个发布的模型都会仔细评估"。

Auto-review 的分类器经过调整专门处理前两种（over-eager 和 honest mistakes），但防御机制适用于所有场景。

---

## 关键的否定设计：拒绝不等于停止

Auto-review 拒绝操作后不会直接停止，而是给 Codex 提供继续的路径：

> "即使 Auto-review 拒绝了一个操作，Codex 通常会自行恢复——找到更安全的推进方式。"

在拒绝后，Codex 在超过一半的情况下能找到可接受的替代方案继续工作。此外，OpenAI 还会监控是否存在"Codex 试图欺骗 Auto-review"的情况：反复拒绝后会自动停止轨迹。

---

## 局限性：这不是确定性安全

OpenAI 明确指出了 Auto-review 的边界：

> "Auto-review 不应被视为安全保证。在自动化和人工红队测试中，我们发现了 Auto-review 可能被误导而批准未经用户批准的命令的案例。我们正在积极修复这些问题，但我们不期望这类系统成为确定性保证的来源。"

此外，Auto-review 不是针对 Model Scheming 的机制：

> "Auto-review 在升级点评估提议的沙箱外操作。一个战略性地隐藏其意图的模型可能仍然产生看起来良性的升级请求，或者可能在 Auto-review 从未看到的沙箱内采取行动。这些风险需要互补的监控和评估方法，包括链式思维监控。"

这是务实的工程判断：Auto-review 不是银弹，但它在实践中大幅提升了安全性，同时保持了可用性。

---

## 对 Agent 工程的意义

Auto-review 代表了一种新兴的 Agent 安全范式：**用对齐的 Agent 监督自主的 Agent**。它不是规则引擎的简单扩展，而是将 LLM 的判断能力嵌入安全关键路径。

这种设计的深层逻辑是：随着模型能力的提升，规则系统无法覆盖所有场景，但另一个对齐良好的模型可以。审查者和执行者的分离，使得安全策略的评估、监控和改进成为可能——而不是把决策权交给一个被优化为"完成任务"的模型。

对于企业级 Agent 部署，Auto-review 的范式提供了重要参考：**安全不是通过限制功能实现，而是通过智能判断实现**。在 Full Access 和完全人工审批之间，存在着大量实际可用的空间。
