# Claude Code 质量回退复盘：Harness 工程的三个致命教训

> 原文：https://www.anthropic.com/engineering/april-23-postmortem  
> Published Apr 23, 2026 | Anthropic Engineering Blog

2026 年 4 月，Claude Code 用户反馈响应质量下降。Anthropic 调查后追踪到三个独立变更同时作用于不同流量切片，最终呈现出广泛、不一致的降级现象。

这不是单一 Bug，而是一场多因素叠加的系统性失效。

---

## 一、三个变更，三种失败模式

### 1. 推理努力默认值：从 high 降到 medium

**背景**：Opus 4.6 发布时，默认推理努力设为 high。但用户反馈 high 模式下偶尔等待时间过长，UI 看起来像冻结了。

**决策**：内部评估显示 medium 努力在大多数任务上略低智能但显著降低延迟，且避免了长尾延迟问题，有助于最大化用户的使用限额。

**结果**：回退耗时过长——用户明确表示愿意为更高智能接受更高延迟。2026 年 4 月 7 日回退。

**教训**：推理努力（effort）是一种权衡——更多思考 vs 更低延迟和使用限额。当产品层选择默认值时，这个选择直接影响用户体验的智能水平。错误默认值的影响面远超单个功能。

> "用户告诉我们，他们更愿意默认更高智能，选择低努力处理简单任务。"

### 2. 缓存优化 Bug：跨层交互的经典案例

**背景**：团队使用 prompt caching 降低用户成本。设计很简单：如果会话空闲超过一小时，清除旧的推理内容（thinking sections）以减少恢复时的未缓存 token 数。

**实现**：使用 `clear_thinking_20251015` API header 配合 `keep:1`。

**Bug**：逻辑错误导致每轮都在清除，而非仅在恢复时清除一次。跨idle阈值后，后续每个请求都会丢弃更早的推理块——包括当前轮次的推理。这导致了「遗忘、重复、奇怪的工具选择」。

**复杂性来源**：
1. 两个不相关的内部实验（message queuing + thinking 显示逻辑）掩盖了 bug
2. 角落情况（stale sessions）难以复现
3. Bug 在 Claude Code 的 context 管理、Anthropic API 和 extended thinking 的交汇处

**发现方式**：回溯测试（back-testing）——用 Opus 4.7 配合必要的代码仓库上下文审查 PR 时发现了 bug，Opus 4.6 则未能发现。

**教训**：跨层交互的 Bug 躲过了所有标准测试流程（unit tests、end-to-end tests、automated verification、dogfooding），因为每个测试层只覆盖了自己的职责范围。这类 Bug 的发现需要「提供必要上下文让模型完整理解代码仓库」。

> "这个 bug 处于 Claude Code 的 context 管理、Anthropic API 和 extended thinking 的交汇处。它通过了多次人工和自动化代码审查。"

### 3. 系统提示词变更：字面意思 vs 实际效果

**背景**：Claude Opus 4.7 有个行为特点——比前代更啰嗦。这在困难问题上更聪明，但也产生更多输出 token。

**变更内容**：
```
"Length limits: keep text between tool calls to ≤25 words. 
Keep final responses to ≤100 words unless the task requires more detail."
```

**测试**：内部测试多周，未见回归。

**问题**：在更广泛的评估集中，Opus 4.6 和 4.7 都出现 3% 的下降。回退后立即恢复。

**教训**：系统提示词的每一行都会影响模型行为。当「减少冗长」和「保持智能」冲突时，字面效果和实际影响完全不同。更重要的是，这种交互效应只在广泛的评估套件下才能发现——单一评估集无法覆盖所有场景。

> "更广泛的评估集显示 Opus 4.6 和 4.7 都下降了 3%。我们立即回退了提示词。"

## 二、三条工程原则

### 原则 1：配置决策是一种工程承诺

将默认推理努力从 high 改为 medium 看起来是配置调整，实际是智能水平的选择。配置变更的「无害感」掩盖了它对用户体验的实质影响。

**工程建议**：任何影响模型行为选择的配置变更，都应该像功能发布一样严格对待——需要评估、 soak period 和回滚预案。

### 原则 2：跨层交互 Bug 是最难发现的 Bug

这个案例的 bug 处于 Claude Code context 管理、API 行为和 extended thinking 的三叉路口。单层测试无法发现跨层问题。

**工程建议**：
- 增加「代码审查时提供完整上下文」的流程，让更强模型发现普通模型漏掉的角落案例
- 建立更完整的 evaluation suite 来覆盖跨层交互场景

### 原则 3：系统提示词变更需要广泛的评估集

`≤25 words` 的字面意思很清晰，但实际效果是智能下降。这说明系统提示词的影响是情境相关的——在某些评估集上通过不代表在其他场景下安全。

**工程建议**：
- 系统提示词变更需要「per-model eval」覆盖
- 增加 ablation（消融）来理解每行的影响
- 添加更长的 soak period 和 gradual rollout

## 三、用户反馈驱动的回退机制

Anthropic 在这次事件中承认：用户通过 `/feedback` 命令分享的问题最终帮助定位了问题。内部 usage 数据和 evals 在早期未能复现用户报告的问题。

这说明：**生产环境的广泛用户反馈是 agent 系统最后也是最准确的评估层**。

用户反馈 → 内部调查 → 根因定位 → 修复 → 经验积累，这个循环的效率决定了系统改进的速度。

---

## 总结

这次复盘揭示的三个失败模式，在任何复杂的 AI Agent 系统中都会反复出现：

| 失败类型 | 根本原因 | 检测难度 | 防范策略 |
|---------|---------|---------|---------|
| 配置决策失误 | 智能 vs 延迟权衡未充分验证 | 中 | 延长 soak period + 用户研究 |
| 跨层交互 Bug | 多系统边界行为组合的意外 | 高 | 更完整的上下文提供 + 跨层 eval |
| 系统提示词副作用 | 提示词交互效应超出单层理解 | 中 | 广泛评估集 + ablation 分析 |

当 agent 系统变得越来越复杂，每一个「优化」都可能引入新的风险。Harness 工程的本质，是在这条线上找到正确的平衡点。

> "这不是用户应该从 Claude Code 获得的体验。作为补偿，我们重置了所有订阅者的使用限额。"
