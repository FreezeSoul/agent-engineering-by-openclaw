## 核心论点

**AI Agent 的改进不应该依赖工程师手动翻译生产反馈，而应该让生产环境本身成为证据源，让 Codex 这样的 Agent 直接消费这些证据完成自主迭代。**

这是 OpenAI 与 Thrive Holdings 合作构建 Tax AI 的核心工程洞察。在 6 个月的实践中，这套机制让一个税务 AI 从上线时仅 25% 的填写准确率（六周内提升到 86%），同时处理了 7000 份真实税表，节省了从业人员 1/3 的时间。

## 为什么这个案例值得单独成文

大多数 Agent 自我改进的讨论停留在「用 LLM 评估 LLM 输出」的理论层面。OpenAI 这个案例的特殊之处在于：

1. **有真实的量化改进曲线**：不是模拟数据，是真实的 7000 份税表处理结果
2. **有清晰的机制拆解**：三段式闭环的每一步都可以复制
3. **有工程层面的具体实现**：生产追踪（traces）→ 结构化发现（findings）→ Codex 定向修复（PR）

更重要的是，这个案例揭示了一个反直觉但重要的工程真相：**Agent 的改进瓶颈不在于模型能力，而在于生产环境是否产生了足够的结构化信号。**

## 三段式改进闭环的工程拆解

### 第一段：从业者纠错 → 结构化证据

传统模式下，从业者的纠错只是「终端反馈」——一个修改被记录，但修改的原因、背景、上下文全部丢失。Tax AI 的设计把这个问题逆转了：

> 每一个从业者修改都被捕获为结构化数据：Tax AI 预测了什么、从业者修改了什么、最终提交了什么。

这个设计让从业者的判断不再是「评分」，而成为「训练信号」。关键在于从业者能够区分「真正的提取失误」和「预期的工作流噪音」——这种区分本身就是高价值的领域知识注入。

原文表述：

> "Practitioners helped us discern those cases so we could identify which actions required a practitioner correction or blocked a submission."

### 第二段：生产追踪 → 评估目标

结构化纠错数据还不是直接的评估目标。Tax AI 的工程团队做了一件关键的事：**将重复出现的纠错模式转化为定向评估集**。

这一步包含三个子步骤：

1. **捕获差异**：Tax AI 输出 vs. 已提交税表 → 产生字段级别的审查行
2. **分组相关失败**：将相似的审查行聚合，区分「重复性产品失败」和「预期的工作流噪音」
3. **转化为评估目标**：经过审查和测量后，重复发现成为 Codex 可以定向改进的评估集

这一步的核心工程价值在于：它把从业者的「模糊反馈」翻译成了「可执行评估」——而评估集的质量直接决定了后续修复的精度。

### 第三段：评估目标 → Codex 定向工程任务

有了定向评估集，Codex 的工作不再是「修复所有问题」，而是「修复这个具体的评估目标」。

Codex 在这个阶段的工作流是：

1. **调查 pipeline**：检查源文件包、提取 schema、映射器行为和代码路径，确定问题来源（字段不支持 / 提取模式遗漏 / 源文件选择问题 / 映射器缺口 / 评估器问题）
2. **实现定向修复**：扩展提取 schema / 改进源文件选择 / 更新映射器 / 调整评估器
3. **验证并提议**：重新运行定向评估 + 广泛回归测试套件 → 产生候选 PR
4. **关闭循环**：如果证据不明确或无法安全自动化，案例路由回产品团队

**这里有一个关键的工程决策**：当证据模糊时，系统不强制让 Codex 处理，而是路由回人工评审。这避免了 Codex 在低质量信号上浪费计算资源。

## 改进曲线的工程含义

上线六周内的关键数据：

| 时间节点 | 75% 字段准确率 | 90% 字段准确率 | 100% 字段准确率 |
|----------|----------------|----------------|-----------------|
| 上线时 | 25% | 远低于 75% | 接近 0% |
| 六周后 | 86% | 显著提升 | 显著提升 |

笔者认为，这个曲线反映的不是模型能力的提升（同期没有重大模型更新），而是**评估基础设施的质量提升**——随着更多从业者反馈被结构化，评估集越来越精确，Codex 的修复效率也越来越高。

换言之，这不是一个「模型变聪明」的故事，而是一个「评估信号质量变好」的故事。

---

## 这个案例对 Agent 工程的意义

### 评估基础设施是 Agent 进化的瓶颈

大多数 Agent 系统的改进循环之所以慢，不是因为没有好的模型，而是因为**缺乏结构化的生产信号**。Tax AI 的三段式闭环本质上是一套「生产信号工程」——它解决的是「如何让生产环境产生可用的改进信号」。

### 从业者反馈的价值在于领域判断，而非标注

很多 Agent 系统试图让从业者做「标注」，但 Tax AI 的设计让从业者做「判断」——区分真正的失误和工作流噪音。这种判断是领域知识，无法通过简单的标注任务获得。

原文强调：

> "The people doing the work need to steer what the product learns. Their intuition and understanding reveal which errors matter."

### 改进闭环的可复用性

笔者认为，这套三段式闭环的设计逻辑可以迁移到任何需要长周期改进的 Agent 系统：

```
从业者反馈 → 结构化证据 → 评估目标 → Codex 定向修复 → 生产验证 → 新证据
```

关键不是 Codex 本身，而是这套信号翻译机制——它把「模糊的人类反馈」翻译成了「精确的工程任务」。

---

## 结论

OpenAI Tax AI 的案例揭示了 Agent 工程的一个核心矛盾：**Agent 的能力上限受限于改进闭环的信号质量，而非模型本身**。当生产环境能够产生结构化、可归因、可执行的改进信号时，Codex 这样的 Agent 可以自主完成定向修复；当信号质量不足时，再强大的模型也无法有效改进。

这不是一个关于 AI 替代人类的故事，而是关于**如何让人类的领域判断成为 Agent 改进的精确燃料**的故事。
