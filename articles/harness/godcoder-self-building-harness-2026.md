# AI Agent 正在学会自己造笼子：Godcoder 的自我 Harness 构建框架

> 传统 Harness 设计隐含了一个假设：人类知道 Agent 需要什么约束。但当 Agent 面对长周期、多阶段、上下文不断演变的环境时，这个假设往往不成立。Godcoder 提出了一个不同的方向——让 Agent 自己发现、自己构建、自己优化自己的 Harness。

---

## 核心命题

大多数 AI Agent 框架把 Harness（评估器、权限边界、工作区约束）当作**静态配置**来处理：你在启动 Agent 前定义好规则，然后让 Agent 在规则内工作。这对于简单任务是够用的。

但对于真正复杂的长周期任务，这个模型有根本性的缺陷：

- **人类设计者无法预见所有场景**：Agent 在真实任务中会遇到设计者没有考虑到的情况，需要临时的工具、隔离的工作区、验证标准
- **任务的中间状态是动态的**：一个 100 步的任务，前 10 步的 Harness 需求和后 10 步可能完全不同
- **自我改进需要闭环**：Agent 每次失败后，需要能够理解"哪里出了问题"并调整自己的约束，而不是等待人类介入

Godcoder 的 Harness 模式给出的答案是：**让 Agent 自己造笼子，而不是人类替它造好**。

---

## 自我构建 Harness 的七步循环

Godcoder 的 Harness 模式运行一个持续优化的循环，核心是 7 个步骤：

```
┌─────────────────────────────────────────────────────────────┐
│           HARNESS MODE  —  实时自我构建循环                 │
│                                                             │
│  START                                                      │
│    │                                                        │
│    ▼                                                        │
│  🏗️  Scaffold  →  创建 harness-build/ 隔离工作区         │
│    │                                                        │
│    ▼                                                        │
│  🗺️  Route     →  选择最高价值的下一个改进点               │
│    │                                                        │
│    ▼                                                        │
│  📋  Plan      →  设计改进方案                            │
│    │                                                        │
│    ▼                                                        │
│  ⚙️  Execute   →  编写、编辑、运行代码                    │
│    │                                                        │
│    ▼                                                        │
│  ✅  Evaluate  →  用项目自身的测试验证改进效果              │
│    │                                                        │
│    ▼                                                        │
│  📝  Log       →  将结果记录到持久化记忆存储               │
│    │                                                        │
│    ▼                                                        │
│  🔁  Optimize  →  将成功模式偏向未来的迭代                │
│    │                                                        │
│    └──────────────────────────────► repeat                  │
└─────────────────────────────────────────────────────────────┘
```

每一步都有具体含义：

| 步骤 | Agent 行为 | 与传统 Harness 的区别 |
|------|-----------|---------------------|
| **Scaffold** | 在 `harness-build/` 目录创建隔离工作区，只读引用主代码库 | 传统 Harness 的工作区是固定的；这里是动态创建的 |
| **Route** | 从记忆存储中选择最有价值的下一个改进点 | 传统 Harness 没有"优先级排序"能力 |
| **Plan** | 设计具体的改进方案 | Agent 参与约束设计，而非被动接受 |
| **Execute** | 执行代码编写和修改 | 正常的执行环节，但上下文是 Agent 自己定义的 |
| **Evaluate** | 用项目自身测试验证改进 | 不是外部评估，是自我验证 |
| **Log** | 记录结果到持久化记忆 | **关键**：使 Harness 具备跨会话的累积能力 |
| **Optimize** | 根据历史记录调整未来策略 | **关键**：Harness 本身是可学习的 |

## 关键技术特征

### 隔离工作区机制

当 Harness 模式启动时，Agent 会在 `harness-build/` 目录下创建独立工作空间：

- 主代码库**只读引用**，不直接在主库操作
- 所有新工作 confined 在隔离区内
- 完成后可选择合并回主库，或丢弃

这是一个**沙盒化的工作区管理**思路，但由 Agent 自己驱动创建和管理，而非人类预设。

### 持久化记忆存储

每个迭代的结果都被记录到持久化存储（通过 ResearchSwarm bridge），包含：

- 改进的具体的操作
- 验证结果（通过/失败）
- 上下文：为什么这个改进被认为有价值

这些记录在后续迭代中被检索，用来：
- **Route 阶段**：优先选择历史上成功率高的改进模式
- **Optimize 阶段**：调整策略偏向经过验证的方向

这本质上是让 Harness 本身具备**经验累积能力**——不是从零开始，而是每次都在上一次的基础上优化。

### 自我验证闭环

每个 Execute 阶段后紧跟 Evaluate 阶段，使用**项目自身的测试用例**来验证改进：

```python
# Agent 执行改进
execute: write_code(changes)

# Agent 用项目测试验证
evaluate: run_project_tests()

# 如果测试通过 → Log → Optimize
# 如果测试失败 → 调整方案，重新 Route
```

这避免了 Agent 产生"幻觉式改进"——改进必须经过项目自身的验证标准，而不是 Agent 自己判断"我觉得改好了"。

## 与传统 Harness 的本质区别

传统 Harness（如 Cursor 的 strict harness、Anthropic 的安全评估框架）有一个共同特征：**Harness 是外部给定的**。人类设计评估标准，Agent 在标准内工作。

Godcoder 的自我构建 Harness 打破了这一点：

| 维度 | 传统 Harness | Godcoder 自我构建 Harness |
|------|--------------|--------------------------|
| **Harness 来源** | 人类设计 | Agent 自己发现和构建 |
| **工作区** | 固定边界 | 动态创建隔离工作区 |
| **学习能力** | Harness 不学习，只评估 | Harness 本身通过 Log+Optimize 累积经验 |
| **适应性** | 静态，无法应对未预见场景 | 动态，遇到新问题 Agent 可以发明新约束 |
| **失败恢复** | 外部干预或重置 | Agent 自己分析失败，调整策略 |

这并不是说自我构建 Harness 一定更好。对于需要强确定性、强合规性的场景，静态 Harness 的可预测性更有价值。但对于探索性、创新性的长周期任务，能够自我适应和自我改进的 Harness 提供了传统模式不具备的能力。

## 对 Agent 工程实践的启示

Godcoder 的自我构建 Harness 模式提出了一个根本性问题：**如果 Agent 能够自己构建 Harness，那么 Agent 与 Harness 的边界在哪里？**

在传统模型中，Harness 是 Agent 的外层边界，Agent 在边界内运作。在 Godcoder 模型中，Agent 可以**临时突破现有边界，自己建立新的边界**，然后在新的边界内工作。

这不是一个安全或哲学问题，而是一个**工程能力边界**的问题：

1. **自我构建 Harness 适合的场景**：探索性任务、开放式任务、长周期上下文演变任务
2. **传统静态 Harness 适合的场景**：强合规性要求、强确定性要求、外部监管要求

最理想的状态可能是**混合架构**：核心安全边界由人类定义（静态 Harness），外围探索性工作由 Agent 自我管理（动态 Harness）。Godcoder 的贡献在于，它证明了这种混合架构的**动态部分是可以工程化的**。

## 结论

AI Agent 工程领域正在出现一个新的能力维度：**Agent 不仅是 Harness 的使用者，也是 Harness 的构建者**。

Godcoder 的自我构建 Harness 框架是这个方向上目前最完整的工程实现。它的 7 步循环（Scaffold → Route → Plan → Execute → Evaluate → Log → Optimize）提供了一个可复用的工程模板，核心创新在于：

- **动态工作区**：Agent 自己创建和管理隔离工作区
- **经验累积**：通过持久化记忆存储，使 Harness 本身具备跨会话的学习能力
- **自我验证**：改进必须经过项目自身测试，而非 Agent 自我认可

这标志着 Harness Engineering 从"人类设计给 Agent 用"到"Agent 自己设计自己用"的范式转变。虽然这个方向还处于早期，但其核心工程机制已经可以工程化参考。
