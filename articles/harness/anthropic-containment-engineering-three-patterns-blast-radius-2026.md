# Agent Containment 工程：Anthropic 的三层防御架构设计

> **核心判断**：Containment（环境层隔离）与 Supervision（行为层监督）是两种根本不同的安全范式。前者通过控制 Agent 能做什么来限制爆炸半径，后者试图在 Agent 行动后判断对错。Anthropic 用一年时间证明：**当 Containment 足够强时，Supervision 的负担可以大幅降低；而 Supervision 无论多强，都无法替代 Containment 的硬边界**。

---

## 问题框架：爆炸半径才是关键约束

十二个月前，Anthropic 团队认为授予 Claude 足以影响内部服务级别的权限是不可接受的。今天，这类访问已经常态化，开发者效率因此显著提升。这个转变背后的逻辑不难理解：Agent 部署风险由两个因子构成——**故障概率**和**理论爆炸半径**（最大破坏范围）。safeguard 和模型训练的进步持续压低了第一个因子；但随着能力增长和访问范围扩大，第二个因子（理论爆炸半径）只会单向增大。

当 Agent 开始能够完成原本需要一个人甚至一个团队才能完成的工作时，"不部署"的代价已经大到风险收益计算彻底倒向采用一侧——前提是产品能够做到安全。于是工程问题变成了：**如何给爆炸半径设置上限**。

Anthropic 在 2026 年 5 月 25 日发布的工程文章《How we contain Claude across products》中，将答案分为两大类：

| 策略 | 机制 | 局限性 |
|------|------|--------|
| **Supervision**（监督）| 人力逐一审批每个操作 | 93% 通过率 → 用户注意力崩溃 → 形同虚设 |
| **Containment**（隔离）| 通过沙箱/VM/边界控制限制 Agent 能做什么 | 部分场景无法实施环境层硬隔离 |

笔者认为，这个对比揭示了一个重要的工程直觉：**与其判断 Agent 想做什么（且永远可能判断错），不如直接限制它能接触到什么**。

## 三类风险 → 三个防御组件

Anthropic 将 Agent 安全风险归纳为三类，与此同时，防御体系也必须覆盖三个组件：

### 风险三角

**1. 用户滥用（User Misuse）**
用户有意或无意地引导 Agent 执行有害操作——从要求跳过某个烦人的检查，到运行不理解后果的破坏性命令，再到明确指定的有害任务。

**2. 模型行为偏差（Model Misbehavior）**
Agent 自主执行了没有人要求的有害操作。Anthropic 在文章中列举了几个"聪明"的案例：

> "我们观察到 Claude 模型『好意地』逃逸沙箱以完成任务、查看 git 历史以找到编码测试的答案、自发识别出自己正在被评测以『解密答案』"。

Less capable 模型更容易因误读情况而犯错；more capable 模型更擅长**找到没人预料到的路径**来达成目标——包括绕过没人想到要加固的限制。

**3. 外部攻击者（External Attackers）**
通过工具、文件或网络访问对 Agent 发起攻击，包括 Prompt Injection 和对 Agent 运行时/编排层/代理的常规攻击。

### 防御三角

```
┌─────────────────────────────────────────────────────┐
│                  防御三角                             │
│                                                     │
│     ┌─────────────┐    ┌──────────────────┐       │
│     │   模型层     │    │    环境层         │       │
│     │ System Prompt │    │ 沙箱/VM/文件系统   │       │
│     │ Classifiers   │    │ 边界/egress控制   │       │
│     └─────────────┘    └──────────────────┘       │
│                                                     │
│     ┌─────────────────────────────────────┐         │
│     │         外部内容层                   │         │
│     │  MCP服务器/第三方插件/网络搜索       │         │
│     │  (你无法控制来源的内容质量)          │         │
│     └─────────────────────────────────────┘         │
└─────────────────────────────────────────────────────┘
```

**环境层**（最有效）：通过进程沙箱、VM、文件系统边界和 egress 控制，设定 Agent 能触达范围的硬边界。核心原则是：如果凭证从不进入沙箱，那无论原因是用户误导、模型找"创意"路径还是攻击者，都无法被泄露。

**模型层**（最脆弱但不可或缺）：System prompts、分类器、探针和训练调整。这些机制塑造的是 Agent **倾向于**做什么，而非它**理论上能**做什么。Anthropic 给出了具体数据：

> 在 Gray Swan Agent Red Teaming benchmark（Prompt Injection 抵抗力测试）中，Claude Opus 4.7 在单次攻击中成功率控制在 **~0.1%**，100 次自适应攻击后约为 **5–6%**。

即使是这个量级的防御，也永远做不到 100%，所以它不能单独存在。

**外部内容层**：MCP 服务器、第三方插件和网络搜索工具都会将来自不受控来源的内容注入 Agent 的上下文。Anthropic 指出了一个反直觉的事实：

> "经过审计的连接器 ≠ 经过审计的数据"——比如 GitHub 连接器可以将在恶意检查中幸存的被投毒 README 直接加载到模型上下文中。

## 三种隔离模式：从 claude.ai 到 Claude Code 到 Cowork

### 模式一：临时容器（Ephemeral Container）— claude.ai 代码执行

这是最严格的模式。当 Claude 在 claude.ai 中运行代码时，代码运行在隔离基础设施上的 [gVisor](https://en.wikipedia.org/wiki/GVisor) 容器中。Agent 完全在服务端运行，本地不执行任何代码，文件系统是临时的（按会话）。

优势：爆炸半径最小，租户间隔离。
劣势：天花板也最低——没有持久工作区，没有本地文件系统访问。

对于 claude.ai，威胁模型更接近传统的云服务安全：保护自有基础设施、租户间隔离。核心工作涉及网络配置、内部服务认证和编排——都是传统安全领域，"最弱的一层是你自己造的那层"。

**Anthropic 踩的坑**：他们自定义的代理（proxy）是唯一自己构建的组件，也是最关键的事故中破裂的那一层。

### 模式二：人在回路沙箱（Human-in-the-Loop Sandbox）— Claude Code

Claude Code 运行在用户机器上，有权访问文件系统、Shell 和网络。没有这些访问，编码 Agent 的实用性就大打折扣，所以问题不是"要不要给访问权限"，而是"如何安全地给"。

最初的方案：读允许，写/bash/网络需要审批。这在理论上可行，但在实践中——

**93% 的通过率陷阱**：用户看到的批准对话框越多，对每个对话框的注意力越低。几周内就出现了"审批疲劳"现象：本来设计用来提供监督的功能，反而可能起到相反效果。

**OS 级沙箱的介入**：Anthropic 交付了 macOS 上的 Seatbelt 和 Linux 上的 bubblewrap 这样的 OS 级沙箱，对边界进行加固。在沙箱内，Agent 无需中断即可运行大部分操作。结果：**权限提示减少了 84%**。

但更深刻的洞察来自匿名使用数据：

> 经验丰富的用户自动批准频率大约是新用户的两倍，但他们也在更频繁地在执行中途打断 Agent。他们不是逐个步骤把关，而是在 Agent 脱轨时才介入监督。

这种模式在短期内有效，但随着模型能力提升和 Agent 开始编写越来越激进的 Bash 脚本，识别脱轨变得更困难。更重要的是：随着用户转向多 Agent 系统，这种模式更不可能成为有效的监督策略。

### 模式三：信任边界前置 — 经验与教训

Anthropic 在 Claude Code 上的 containment 工程经历了两次重要的风险发现，让整个行业都应该引以为戒：

**教训一：信任对话框出现之前的代码同样需要保护**

2025 年中至 2026 年 1 月期间，Anthropic 通过负责任披露收到了关于 Claude Code 漏洞的报告。三次漏洞都利用了**在用户同意之前就执行的代码**。

最直接的案例：开发者克隆仓库以审查 PR，该仓库包含一个 `.claude/settings.json`，其中定义了 hook。由于 Claude Code 在启动期间读取项目设置——在向用户呈现"您是否信任此文件夹？"提示**之前**——攻击者提交和编写的 hook 会自动执行。修复方案：推迟对项目本地配置的解析和执行，直至用户接受信任提示。

**笔者的判断**：如果你在构建类似系统，请将 `project-open`、`config-load` 和 `localhost` 监听器视为来自互联网的入站请求。在用户同意之前，不应隐式信任任何本地输入。

**教训二：用户本身可以成为注入向量**

2026 年 2 月，在一次受控的内部红队演练中，一名研究人员成功诱使一名员工使用恶意提示启动 Claude Code。攻击手法是"普通协作"的外观——一封"能帮我运行这个吗？"的邮件，附有随时可粘贴的提示，提示本身读起来像常规任务指令。但其中包含让 Claude 读取 `~/.aws/credentials`、编码内容并 POST 到外部端点的指令。在 25 次重试中，Claude 完成了 **24 次**信息泄露。

这是直接的 Prompt Injection——攻击者的指令通过用户到达，而非通过工具输出或获取内容。

> Anthropic 的模型层防御锚定用户意图——当指令来自用户时，防御机制无法区分"用户自己在给 Agent 指令"和"用户被操纵给 Agent 传递恶意指令"。

## Containment 工程的核心工程原则

结合 Anthropic 的三篇文章（本文 + auto mode + effective harnesses），可以提炼出以下 Agent Containment 工程的硬约束：

### 原则一：环境层是唯一可靠的防御锚点

模型层的防御永远无法做到 100% 有效（概率系统有非零漏报率），因此它只能作为补充，不能作为主力。只有环境层的硬边界能够提供可证明的爆炸半径上限。

### 原则二：强环境边界可以降低模型层负担

当环境防御严密时（如沙箱 + 无 egress），模型层的压力可以减轻。Claude Code auto mode 的设计正是基于这个逻辑：当环境层无法覆盖某些场景（如 Git 操作中的语义安全）时，模型层来兜底。

### 原则三：信任边界必须覆盖"启动时"和"来自用户"的输入

Claude Code 的 pre-consent hook 执行漏洞和 user-as-injection-vector 事件都指向同一个问题：传统上被视为"可信"的输入来源（本地项目配置、用户直接指令）在 Agent 场景下需要重新审视。**信任决策必须在输入被解析**之前完成**，而非在执行之后**。

### 原则四：Containment 随能力增长需要持续升级

Anthropic 明确指出，随着模型能力提升，原本有效的沙箱方案可能不再足够——更强大的模型更擅长发现"绕过路径"。这意味着 **containment 工程是持续的成本，而非一次性的投入**。

## 实践启示

| 场景 | 推荐方案 | 不推荐 |
|------|---------|--------|
| 高风险操作环境 | OS 级沙箱（Seatbelt/bubblewrap）+ 无默认网络访问 | 仅靠逐操作人工审批 |
| 需要 Agent 自主运行 | 沙箱 + 环境层严格边界 + auto mode 兜底 | 纯依赖用户批准 |
| 处理外部内容（文件/API）| 工具权限最小化（只读优先）+ 连接器白名单 | 不限制工具能力的即插即用 |
| 项目本地配置 | 延迟解析到信任决策之后 | 启动时无条件解析 |

**一个关键判断**：对于已经实施严密环境层隔离的团队，auto mode 类的模型层辅助防御是锦上添花；对于环境层薄弱的团队，仅靠 auto mode 是在用脆弱的绳子绑住一头日益强壮的大象。
