# OpenAI MRC 超级计算网络：如何用 SRv6 源路由支撑 10 万 GPU 集群

> **来源**：https://openai.com/index/mrc-supercomputer-networking/（May 5, 2026，Engineering Blog）
> **核心判断**：MRC = Multi-path Reliable Connection，一个让 10 万+ GPU 同步训练成为可能的网络协议

---

## 背景：为什么 AI 训练需要新的网络设计

当训练大规模 AI 模型时，单个训练步骤可能涉及数百万次数据传输。**任何一次传输延迟，都会像涟漪一样扩散到整个任务**，导致 GPU 处于空闲状态。

传统网络的问题在集群规模扩大时急剧恶化：

1. **拥塞不可避免**：两个 GPU 同时向同一目的地发送数据时会产生瓶颈
2. **单点故障代价高昂**：一个链路故障往往导致整个训练任务崩溃，必须从检查点重启，或停滞数秒等待网络重新计算路由
3. **同步预训练是「故障放大器」**：在同步预训练中，许多 GPU 跨多台计算机以锁步方式协作训练一个模型，**任何单一链路抖动都会被放大**

OpenAI 的 Stargate 超级计算机需要解决这个问题。

---

## MRC 的核心设计：三招解决网络可靠性

### 第一招：多平面网络（Multi-plane Networks）

**不是把一个 800Gb/s 链路当作一条路，而是把它分成多条小路。**

例如，一个网络接口可以连接到 8 个不同的交换机。这样可以构建 8 个独立的并行网络（或称「平面」），每个平面以 100Gb/s 运行，而不是一个 800Gb/s 的单一网络。

**效果**：
- 一个 64 端口 800Gb/s 的交换机可以改为连接 512 个端口（每个 100Gb/s）
- 约 131,000 个 GPU 只需两级交换机即可全连接
- 传统 800Gb/s 网络需要三到四级交换机

这降低了功耗、减少了可能发生故障的组件数量，降低了总体网络成本，同时提供更多的路径多样性。

### 第二招：数据包喷雾（Packet Spraying）

**传统方法**：每个传输走单一路径，数据包按顺序到达 → 容易拥塞，路径利用率低

**MRC 的方法**：将单个传输的数据包喷雾到数百条路径上，跨所有平面分布传输

工作原理：
- MRC 连接的每个数据包可以走不同的路径，**可能无序到达**
- 但所有 MRC 数据包都包含最终内存地址，目的地可以在到达时直接交付到内存
- 通过在多条路径上分布流量，MRC 避免了网络的热点区域，防止某些事务比其他事务花费更长时间

**自适应负载均衡**：如果 MRC 检测到某条路径变得拥塞，它会立即换到另一条路径，均匀分配网络负载。

**拥塞修剪（Packet Trimming）**：如果交换机因拥塞需要丢弃数据包，它会修剪掉有效载荷，只转发 header 到目的地，触发明确的重新传输请求。这减少了误报——避免错误地将丢包归因于路径故障。

### 第三招：静态源路由（Source Routing with SRv6）

**传统网络**：交换机运行动态路由协议（如 BGP），计算可用路径并在故障时重新路由 → 复杂、可能失败

**MRC 的方法**：禁用动态路由，使用 IPv6 Segment Routing（SRv6）

SRv6 让发送者直接在数据包的**目标地址中嵌入路径序列**（经过的交换机标识符列表）。

工作原理：
1. 转发时，交换机检查自己的标识符是否存在于数据包中
2. 如果存在，通过移动目标地址来移除标识符，揭示下一个交换机的标识符
3. 交换机在静态路由表中查找该标识符，确定下一跳发送位置
4. 这个静态路由表在交换机首次配置时设置，**此后永不更改**

关键结果：如果路径丢包，MRC 停止使用该路径。**交换机不需要重新计算路由或做任何其他事情——它们只是盲目遵循配置好的静态路由。**

## 生产验证：Stargate 超级计算机的真实表现

MRC 已经部署在 OpenAI 最大的 NVIDIA GB200 超级计算机上，包括 Oracle Cloud Infrastructure（OCI）在德克萨斯州阿比林的站点，以及微软的 Fairwater 超级计算机。

**实测数据**：

- 在一次前沿模型（用于 ChatGPT 和 Codex）的训练过程中，团队不得不重启四台一级交换机
  - **之前**：重启交换机需要运营团队非常小心，避免中断训练
  - **现在**：甚至不需要与集群中运行训练任务的团队协调

- 在训练期间，团队观察到在 tier-0 和 tier-1 交换机之间每分钟有多次链路抖动
  - **MRC 确保它们对同步预训练任务没有可测量的影响**
  - 实际上，影响小到甚至不需要立即优先修复这些链路

- 如果 GPU 网络接口和 tier-0 交换机之间的链路发生故障：
  - **之前**：训练任务会失败
  - **现在**：任务以合理的性能存活。如果 8 端口网络接口失去一个端口，最大速率降低 1/8。MRC 检测到这一点，重新计算路径以避开故障平面，并立即通知对等方不要将该平面用于入站流量。大多数故障链路在一分钟内恢复，此时 MRC 恢复使用该平面

## 关键数据：三个核心优势

| 优势 | 具体表现 |
|------|----------|
| **多平面高密度** | 超过 100,000 个 GPU 只需两级以太网交换机，提供足够的冗余度来度过网络故障，同时功耗低于等效的三级或四级单平面网络 |
| **自适应数据包喷雾** | 网络核心几乎无拥塞，大幅减少同步训练期间流之间的吞吐量变化；多个任务共享集群时互不影响性能 |
| **SRv6 源路由** | 绕过故障快速恢复，运行简单的静态网络控制平面，消除动态路由故障的整类行为 |

## 核心洞察

**MRC 的本质**：将网络可靠性从「尽力而为」变为「可预测」。通过喷雾 + 源路由的组合，MRC 可以在**微秒级**检测网络故障并绕过它们，相比之下，传统网络结构可能需要**秒甚至数十秒**来稳定和绕过故障。

**对 Agent 工程的隐含意义**：虽然 MRC 直接面向 AI 训练基础设施，但它揭示了一个更普遍的原则——**当系统规模足够大时，可用性不再是可选项，而是基础设施的一部分**。对于运行长时间 Agent 任务的分布式系统，这种可靠性设计思想值得借鉴：多平面冗余 + 快速故障隔离 + 静态路径规划，比动态恢复更可靠。
