# Cursor 云端 Agent 构建一年后的六条核心教训

> 原文：["What we've learned building cloud agents"](https://cursor.com/blog/cloud-agent-lessons)，Josh Ma，Cursor Engineering，2026-05-21

Cloud Agent 不只是「把本地 Agent 搬到服务器上」——这是一年后的核心教训。

Cursor 工程团队在 2026 年 5 月的官方博客中披露了构建 Cloud Agents 的真实踩坑历程。他们发现，当云端 Agent 需要长时间运行、跨多台机器并行、承担更复杂任务时，挑战的维度完全变了：环境配置变成产品本身、可靠执行需要 Temporal 这样的工作流引擎、解耦成为架构主线。本文从工程师视角系统梳理这六条核心教训。

---

## 一、开发环境本身就是产品

**本地 Agent 的环境是「免费」的**——它直接复用你笔记本上已经配置好的开发环境。但在云端，**一切都需要从零重建**。

Cursor 团队在这一年里学到：Cloud Agent 输出质量的**最大单一决定因素**是它是否拥有完整的开发环境。但棘手之处在于，缺少环境的表现形式非常隐蔽——不是报错，而是一种「微妙的输出质量下降」。开发者可能以为是模型问题，实际根源在于环境配置不完整。

为了解决「完整环境」问题，Cursor 不得不在基础设施层面构建大量能力：

- 用户工具：构建和管理 Agent 环境配置
- 虚拟机生命周期管理：在消息之间高效地休眠和恢复 Agent 虚拟机
- 镜像检查点/恢复/分叉管道：快速且持久地保存、恢复和分叉 VM 镜像
- Agent 与客户端的紧耦合集成：让人和 Agent 都能理解和交互环境

随着 Cloud Agents 承担更多工作，它们需要**受控的网络访问**来创建 PR、拉取依赖、做调研。Cursor 团队发现自己实际上在**为 Agents 构建企业级 IT 基础设施**——包括密钥编辑、网络策略、凭证管理。

> **笔者认为**：这揭示了一个重要趋势——当 Agent 能力足够强时，「环境配置」从开发者的个人负担变成了平台的核心竞争力。Cursor 的护城河不是模型本身，而是这套「让 Agent 拥有完整开发环境」的基础设施。

---

## 二、长时间运行的 Agent 需要持久化执行

Cloud Agent 带来的可靠性挑战与本地 Agent 本质不同。本地 Agent 在笔记本上竞争资源，而 Cloud Agent 运行在独立隔离的虚拟机中——这让开发者可以并行运行多个 Agent、委托需要数小时甚至数天的长任务。

但 VM 也带来了暴露问题：推理提供商中断、Pod 需要替换、EC2 节点宕机。

Cursor 早期用**工作窃取架构**（work-stealing architecture）：Worker 节点可以捡起 Agent 并循环到完成。这是从本地到服务器的直接移植，结果很脆弱——Cloud Agent 早期测试版的可靠性只有**一个 9**（90%）。

Cursor 在构建 Temporal 工作流引擎能力的过程中，最终发现他们在重建 Temporal 已经解决的很多原语（重试机制、跨机器调度、节点故障下的持久化）。于是他们迁移到了 Temporal。

迁移后，Cloud Agent 循环可以承受：推理可靠性波动、Pod 休眠和恢复、跨越数天甚至数周的运行。**仅这一迁移就让可靠性超过两个 9**。目前 Temporal 每天处理超过 **5000 万次操作**，横跨超过 **700 万个独立工作流**。

Cursor 还学到如何更好地架构 Temporal 工作流：

1. 从「永久」Agent 工作流 → **多个更短的、工作完成即退出的工作流**，这让版本升级更容易
2. 把 Activity 拆分出来，以更好地捕获超时和重试——因为异步工具调用、子 Agent 和推理提供商中断改变了底层假设

> **笔者认为**：这验证了一个工程常识——当你需要「Durable Execution」时，直接用 Temporal 而不是自己造轮子。Cursor 的经验表明，从零构建可靠的长时间运行 Agent 的基础设施成本远高于集成成熟开源工作流引擎的成本。

## 三、解耦 Agent、机器与会话状态

Cloud Agent 不再是「一台机器上运行一个循环」。

实际情况是：一个 Agent 可能在一台机器上运行，跨多台机器生成异步子 Agent，或者先在本地启动然后委托给云端。子 Agent 可能比父 Agent 寿命更长，或者运行在完全不同的 Pod 类型上。

为了实现这种灵活性，Cursor 团队发现把**Agent 循环、机器状态和会话状态作为解耦组件**非常有价值：

- 因为 Agent 循环运行在 Temporal 而不是 VM 本身，他们可以**独立管理 Pod 生命周期**，让 Agent 跨不同类型的 Pod 运行（包括只读 VM 或预热 VM 这样的优化）
- 在会话侧，他们将**存储和流式层**从核心 Agent 工作流中分离出来。构建了高效的追加写存储机制，将会话更新流式传输到 Web 和桌面客户端。这层考虑了重试——如果 Agent 循环的一步在流式传输部分输出后失败然后重试，客户端可以检测到这一点、倒回其流并显示新数据而不是旧数据

> **笔者认为**：这种架构选择的本质是**关注点分离**——Agent 的业务逻辑（循环）、执行环境（VM）、数据状态（会话）本就应该独立演进。Cursor 的教训说明，当你的系统需要支持「Agent 可能比父 Agent 活得久」这样的场景时，架构层面的解耦不再是可选项。

## 四、知道什么时候「让开」

构建 Cloud Agent Harness 意味着不断重新评估：有多少行为是确定性的，多少是交给 Agent 的。

早期 Cursor 对 Agent 不够信任，Harness 会在每个任务完成后双重检查、强制提交和推送。随着模型变得更聪明，他们开始把逻辑从 Harness 移到 **Agent 控制的工具**中。

一个具体例子：多仓库设置一年前还需要硬编码的 Harness 行为。现在可以给 Agent 仓库布局、暴露分支和 PR 的工具，然后让它自己决定怎么做。

同样的演变发生在 CI Autofix：早期版本的 Cloud Agent Harness 包含抓取 Job 失败日志并写入 VM 的逻辑。现在他们只是给 Agent 访问 GitHub CLI，**大输出自动写入它可以搜索的文件**。对 Agent 的通知简单多了，这种趋势预计会持续。

Harness 不会消失，而是**其内容在变化**。Computer Use 就是一个好例子：Cursor 的 Cloud Agent Harness 有专门的子 Agent 类型负责 Computer Use，有自己的模型路由、自定义提示和屏幕录制。VNC 和 Chrome 属于环境，被父 Agent 和子 Agent 共享。这让父 Agent 可以直接利用它们（例如，运行 Playwright 脚本）。这种脚手架之所以必要，是因为模型还没准备好独立处理 Computer Use，但 Agent 仍然控制着何时调用它。

Cloud Agent 的提示词也与本地 Agent 不同。他们鼓励 Cloud Agent 更**自主**，因为阻塞的代价要高得多。本地环境你可以知道 Agent 什么时候停止并等待许可，但在云端它可能在那里坐上几个小时，直到你回去检查。

> **笔者认为**：这是一个「逐步释放控制权」的过程。当 Agent 能力不足时，Harness 需要做更多检查和引导；当 Agent 足够可靠时，这些检查反而成为瓶颈。Cursor 的演进路径表明，**好的 Harness 设计是动态的**——它会随着 Agent 能力增长而收缩，把控制权逐步移交给 Agent。

## 五、自愈的 Agent 环境

Cursor 的未来聚焦方向：超越「保护 Agent」和「放手让 Agent 做」之间的二元选择。

更好的模式是：**给 Agent 理解周围系统的工具**。

他们希望 Cloud Agent 能够：

- 当密钥缺失、网络访问被阻止或其环境阻止其取得进展时**主动报告**
- 然后**以自愈的方式采取行动**

在[最近的 Composer 2.5 研究博客](https://cursor.com/blog/composer-2-5)中，Cursor 讨论了一个具体场景：模型可以自主修复缺失的依赖。这些能力的结合——主动诊断问题 + 自主修复——是 Cursor 眼中「自愈 Agent 环境」的终态。

> **笔者认为**：自愈的本质是**让 Agent 拥有关于环境的完整信息，并让它有能力采取行动**。Cursor 的方向不是让 Harness 变得更复杂，而是让 Agent 本身变得更自主——这是从「防护性编程」到「赋能式设计」的转变。

## 六、关键数据点

| 指标 | 数值 |
|------|------|
| Temporal 每日处理操作数 | 5000 万+ |
| Temporal 独立工作流数 | 700 万+ |
| Cloud Agent 可靠性 | 从 ~90%（1个9）→ 99%+（2个9）|
| Cursor 内部 PR 来自 Cloud Agent | 40%+ |
| 架构演进 | 工作窃取 → Temporal 迁移；永久工作流 → 多个短工作流 |

## 结论：云端 Agent 的本质是「为 Agent 建 IT 基础设施」

Cursor 一年构建 Cloud Agent 的最大收获是：这不是「把本地 Agent 搬到云端」，而是**为 Agent 构建一整套企业级 IT 基础设施**——包括环境配置、可靠执行、网络策略、凭证管理。

这条路的终点是：**让 Agent 拥有完整的开发环境 + 持久的执行能力 + 自愈的故障处理机制**。Cursor 认为，随着模型能力的提升，「环境」将成为决定 Agent 能否发挥全部潜力的关键因素。

这也解释了为什么 Cursor 在基础设施层面投入如此之多——他们的护城河不只是模型，而是**让 Agent 在云端真正有效地工作**的系统能力。
