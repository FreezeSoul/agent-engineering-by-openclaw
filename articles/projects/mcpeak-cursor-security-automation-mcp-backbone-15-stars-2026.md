## 这个项目解决什么问题

Cursor 安全团队在生产环境运行四个安全 Agent，每天处理 3000+ PR，捕获 200+ 真实漏洞。这些 Agent 不是"问一答一"的单体，而是**共享状态、去重、统一输出**的协作系统。

这个仓库公开了支撑这套系统的核心组件：

1. **security-automations-mcp/**：MCP 服务器，处理安全发现的收集、去重和路由
2. **slack-notification-service/**：认证的 Slack 投递服务

**三个 MCP tool 的设计**：

### 1. 持久化数据存储

Agent 每次发现漏洞，通过 MCP tool 写入持久化存储。这让 Cursor 能够：
- 追踪安全影响的时序变化
- 优化触发策略（哪些类型的漏洞应该触发告警）
- 跨运行比对结果

```python
# 伪代码示例（MCP tool interface）
async def store_finding(finding: SecurityFinding) -> str:
    """存储安全发现，返回 ID 用于后续追踪"""
    await db.insert("security_findings", {
        "type": finding.type,
        "severity": finding.severity,
        "file": finding.file,
        "line": finding.line,
        "timestamp": datetime.utcnow(),
        "agent": finding.source_agent,
    })
    return finding_id
```

### 2. 去重（Gemini Flash 2.5 驱动）

多 Agent 并行运行可能用不同措辞描述同一个漏洞。去重分类器判断两个发现是否"语义等价"：

> "We run multiple review agents on every change, and because their findings are generated by an LLM, different agents can end up using different words to describe the same underlying issue."

这个分类器不是简单的字符串匹配，而是**语义等价判断**——这在安全场景中特别重要，因为同一个漏洞可能出现在不同文件、不同措辞的描述中。

### 3. 一致输出路由

所有 Agent 的发现通过统一格式发送到 Slack，支持 dismiss/snooze 操作。这意味着 Slack 消息的结构是**标准化的**，便于安全和合规团队做后续处理。

## 为什么这个项目值得关注

### 1. MCP 的真实生产用法（不是玩具）

很多 MCP 教程讲的是"怎么让 Agent 调用工具"。这个仓库展示的是**怎么用 MCP 实现多 Agent 间的状态共享和协调**。

三个 MCP tool（store / dedup / route）分别对应多 Agent 协作中的三个核心挑战：
- 状态持久化
- 发现去重
- 输出一致性

### 2. Lambda 部署的 Just-in-Time 模式

MCP server 以 **serverless Lambda** 方式部署，只在需要时启动，不占用常驻资源。这是 Agent 基础设施的"绿色"方案——按需扩展，而不是常驻占用。

### 3. Terraform 管理安全工具本身

所有变更通过 Terraform 管理，意味着**安全 Agent 本身的变更有完整的 code review 流程**。这是一个容易被忽略的工程细节：你的安全 Agent 本身安全吗？谁来审计它的变更？
