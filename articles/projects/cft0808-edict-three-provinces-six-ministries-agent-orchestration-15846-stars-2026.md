## 核心洞察

Edict（三省六部）用中国古代帝国政治制度重新设计了多 Agent 协作架构。其核心洞察是：**大多数 Multi-Agent 框架让 Agent 自由协作，结果无法复现、无法审计、无法干预**。而唐代的三省六部制天然实现了权力分立与制衡——Edict 将这一制度迁移到 AI Agent 世界：太子分拣、中书省规划、门下省审核封驳、尚书省派发、六部并行执行。

**制度性审核**是 Edict 区别于 CrewAI、AutoGen、MetaGPT 的核心差异：门下省专职审查中书省的规划方案，不合格直接打回重做，而非可选的 warning。这解决了 Multi-Agent 协作中"做完就交，没有 QA"的根本问题。

## 架构解析

### 十二 Agent 分权制衡

```
用户（皇上）→ 太子（分拣）→ 中书省（规划）→ 门下省（审议）→ 尚书省（派发）→ 六部（并行执行）→ 回奏
```

| Agent 角色 | 职责 | 独特之处 |
|-----------|------|---------|
| 太子 | 消息分拣 | 闲聊自动回复，只将旨意建为任务 |
| 中书省 | 任务规划 | 方案完备性审查 |
| 门下省 | 审核封驳 | **强制质量关卡**，不合格打回重做 |
| 尚书省 | 任务派发 | 权限矩阵校验，非法路径拒绝 |
| 六部 + 吏部 | 并行执行 | 户/礼/兵/刑/工/吏各司其职 |
| 早朝官 | 兼容角色 | 处理边缘场景 |

### 对比主流框架

| | CrewAI | MetaGPT | AutoGen | **三省六部** |
|--|:------:|:-------:|:-------:|:-----------:|
| 审核机制 | ❌ | ⚠️ 可选 | ⚠️ Human-in-loop | **✅ 专职审核·可封驳** |
| 实时看板 | ❌ | ❌ | ❌ | **✅ 军机处 Kanban** |

| Agent 健康监控 | ❌ | ❌ | ❌ | **✅ 心跳+活跃度检测** |
| 热切换模型 | ❌ | ❌ | ❌ | **✅ 看板内一键切换 LLM** |

## 功能全景

### 军机处看板（10 个功能面板）

- **旨意看板**：Kanban 视图 + 部门过滤 + 全文搜索 + 心跳徽章（🟢活跃 🟡停滞 🔴告警）
- **省部调度 Monitor**：各状态任务数量可视化 + Agent 健康实时卡片
- **奏折阁**：已完成任务自动归档为五阶段时间线（圣旨→中书→门下→六部→回奏），一键复制为 Markdown
- **旨库**：9 个预设模板 + 分类筛选 + 预估时间和费用
- **官员总览**：Token 消耗排行榜 + 活跃度统计
- **模型配置**：每个 Agent 独立切换 LLM，~5 秒生效
- **朝堂议政**：多 Agent 围绕议题依职责展开辩论，LLM 驱动多角色讨论

## 技术实现

- **后端**：纯 stdlib（零第三方依赖），避免依赖地狱
- **前端**：React 18 看板
- **部署**：Docker 一键部署（`docker run -p 7891:7891 cft0808/edict`）
- **OpenClaw**：必须依赖，通过 OpenClaw Gateway 实现 Agent 通信
- **权限矩阵**：白纸黑字的权限定义，严格控制谁能给谁发消息
- **状态流转校验**：kanban_update.py 强制合法状态转换路径

## 为什么值得关注

Edict 的贡献不在于发明新技术，而在于**将历史制度迁移到 AI 架构设计**。其"制度性审核"机制直接回应了 Multi-Agent 系统中 QA 缺失的痛点——当 Agent 可以自主执行危险操作时，一个强制性的质量关卡比人类的 final review 更可靠。

对于 AI Coding 和 Agent 系统的工程实践，Edict 提出了一个根本性问题：**我们是否需要为 AI Agent 设计"宪法"来约束其权力？** 这一思路与 Anthropic 的 Constitutional AI 和 Claude Code Auto Mode 的双层防御架构异曲同工。
