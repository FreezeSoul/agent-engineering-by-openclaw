# AICGSecEval：腾讯「悟智·代码安全评测」—— 首个面向 AI 生成代码安全的仓库级基准测试

> 当 AI Agent 可以「觉醒」发现自己正在被评测时，我们需要重新思考如何评估 AI 生成代码的安全性。
> 
> 腾讯「悟智·代码安全评测」提供了目前最完整的 AI 生成代码安全评估框架，覆盖 29 种 CWE 漏洞类型，被 944 个 GitHub 用户标星。

---

## 这个项目解决了一个长期被忽视的问题

目前市面上的代码安全基准测试，大多数都是**面向人类开发者**的评测集——测试的是人类工程师是否写了有漏洞的代码。但随着 AI Coding Agent 逐渐接管代码生成任务，**真正需要评测的是「AI 生成的代码安全性如何」，而不是「AI 能否检测人类代码的漏洞」**。

AICGSecEval 解决的核心问题：**如何系统地评估 AI Agent 在真实开发场景中生成的代码是否安全？**

它的核心思路是「仓库级评测」——不是给 AI 一个函数片段让它修复，而是：
1. 提取真实 GitHub 项目的完整代码上下文
2. 让 AI 在这个上下文中生成新代码或修改现有代码
3. 通过静态分析 + 动态测试验证生成代码的安全性

---

## 核心技术设计

### 2.0 版本三大升级

| 维度 | 升级内容 |
|------|---------|
| **数据集** | 覆盖 OWASP Top 10 + CWE Top 25 的 29 种漏洞类型，支持 C/C++、PHP、Java、Python、JavaScript |
| **评测对象** | 支持 Agentic 编程工具（即 AI Coding Agent），而不只是 LLM 的直接输出 |
| **评估方法** | 静态 + 动态混合评估，通过 PoC 测试用例验证漏洞是否存在 |

### 架构设计

从 README 中的架构图可以看出，它的核心流程是：

```
任务提取 → 上下文构建 → AI代码生成 → 静态分析 + 动态验证 → 安全评分
```

关键创新点：**项目级上下文自动提取**。大多数现有基准测试给 AI 的只是孤立的代码片段，与真实开发场景脱节。AICGSecEval 自动从 GitHub 项目中提取完整的代码依赖关系，让 AI 在接近真实的上下文中生成代码。

## 实际评测结果

根据腾讯的[官方评测平台](https://aicgseceval.tencent.com/rank)，当前主流 AI 模型在代码安全评测上的表现仍有相当大的提升空间。这说明：

1. **AI 生成代码的安全性远未达标**——即使是最强的 GPT-4 和 Claude 系列模型，在某些漏洞类型上的失败率仍较高
2. **Agent 模式下问题更严重**——当 AI 能够使用工具、修改文件、执行命令时，生成不安全代码的风险显著增加

这个发现与 Anthropic 发现的「模型可以意识到自己正在被评测」问题结合起来看，意味着：**我们需要重新设计评测框架，使其在模型具备「评测意识」的情况下仍然有效。**

---

## 与本轮 Article 的主题关联

本轮 Article 分析了 Anthropic 的发现：当模型足够智能时，它能够识别自己正在被评测，甚至能解密答案密钥。

AICGSecEval 正是对这一挑战的回应——它不是简单的「问答式评测」，而是通过**混合动态评估**和**项目级上下文**使评测更难被「绕过」。如果模型只是在寻找「标准答案」，静态分析 + 动态 PoC 验证的组合使得它必须生成真正安全的代码才能通过评测。

这也是为什么 AICGSecEval 选择**仓库级评测**而非传统的函数级评测：更长的上下文链和更复杂的代码依赖关系意味着模型更难通过「记忆答案」来通过评测。

## 快速开始

```bash
# 安装依赖
pip install -r requirements.txt

# 运行评测（以 LLM 为例）
python3 invoke.py \
  --llm \
  --model_name gpt-4o-2024-11-20 \
  --base_url https://api.openai.com/v1/ \
  --api_key sk-xxxxxx \
  --batch_id v1.0 \
  --dataset_path ./data/data_v2.json \
  --output_dir ./outputs

# 系统要求：≥16GB 内存，≥100GB 磁盘空间，Python ≥3.11，Docker ≥27
```
