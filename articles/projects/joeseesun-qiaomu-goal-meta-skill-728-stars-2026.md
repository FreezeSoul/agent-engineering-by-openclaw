# qiaomu-goal-meta-skill：将模糊需求翻译成可执行契约的元技能

## 核心命题

很多 agent 任务失败，不是因为模型不会写代码，而是**目标一开始就太松**——"做得高级一点""帮我开发一个 App""修一下这个问题"，这些话对人类有感觉，对 agent 却缺少几个关键东西：怎么验证、哪里不能碰、失败后怎么迭代、什么时候必须暂停。

qiaomu-goal-meta-skill 的核心价值：**让 Codex 在开工前拿到一份更像"任务合同"的 `/goal`**，而不是一张需要填写的规划表。

## 为什么这个项目值得关注

### 1. 直击 Harness 工程的核心缺陷

当前大多数 agent harness 关注的是"运行时的安全"，但忽略了**启动时的目标质量**。一个缺少验证条件的 `/goal`，会让 agent 在错误的轨道上跑很远才发现白干了；一个没有边界约束的目标，会让 agent 不知不觉越界到付费、账号、生产数据等高风险区域。

qiaomu 从**目标定义层**解决这个问题——不是给 agent 更多的运行时限制，而是给 agent 一个**更清晰的任务合同**。

### 2. 七层结构覆盖了 Harness 的完整闭环

qiaomu 生成的 `/goal` 包含七个关键字段，不是模板填充，而是每一条都解决真实问题：

```
/goal [具体任务描述]
验证：[具体可查证的输出证据]
约束：[绝对不能做的事项]
边界：[只写什么、不写什么的范围]
迭代策略：[失败后怎么调整]
完成条件：[什么情况下算真正完成]
暂停条件：[遇到什么必须停下来等人工]
```

以一个"GTA 网页游戏"为例：

> **推荐执行版（中文，可直接复制）**：
> `/goal 开发一个第一版原创 GTA-like 网页游戏 MVP，用户可以在浏览器中进入一个俯视 2D 城市场景，控制角色步行、上车驾驶、完成一个简单任务，并看到基础状态反馈。`
>
> `验证：如果已有项目，先读取项目脚本和运行方式；否则新建最小网页游戏项目，启动本地开发服务，在浏览器中完整走通一次进入游戏、移动角色、上车驾驶、触发任务、完成任务、查看状态反馈流程...`
>
> `约束：不使用 GTA、Rockstar 或任何真实游戏的名称、Logo、角色、地图、音乐、音效、美术素材...`
>
> `暂停条件：需要使用 GTA 官方 IP 或素材、真实版权素材授权、联机服务器、账号系统、支付...时暂停。`

这段输出里，**暂停条件**是最容易被大多数 agent 忽略的字段，却是 Harness 设计的核心：它明确告诉 agent"遇到这些情况不要自己判断，要停下来"。

### 3. 模糊词不删除，翻译成验证条件

这是 qiaomu 最有价值的工程哲学之一。"高级""有质感""专业""像官网"不是坏词，坏的是把它们当完成标准。qiaomu 会把它们变成：

- 设计方向（而不是设计结论）
- 截图检查（而不是主观评价）
- 最多 3 轮聚焦视觉改进（而不是无限迭代）

这本质上是把**模糊的审美目标翻译成可验证的工程目标**——正是大多数 agent prompt 做不到的事。

### 4. 懒人选择题：不用写长句，回选项就能继续收敛

qiaomu 在给出推荐执行版之后，还提供编号可选调整：

```
可选调整
1. 项目形态：A 新建本地网页游戏 MVP（默认）/ B 改现有项目 / C 先做可交互原型
2. 画面形式：A 俯视 2D（默认）/ B 像素风 2D / C Three.js 3D
3. 第一版范围：A 步行+驾驶+一个任务（默认）/ B 加警戒/追逐系统 / C 加多任务和商店
4. 验证方式：A 本地浏览器桌面+移动检查（默认）/ B 加自动化测试 / C 部署后线上验证
```

这解决了 human-in-the-loop 的效率问题：用户不需要写长句描述，只需要回"按默认"或"1A 2B 3B 4A"，agent 就能继续收敛。

## 技术实现分析

### 架构：单文件 Skill + Linter

qiaomu 是一个标准的 Codex Agent Skill（`SKILL.md`），不需要复杂的框架或依赖。安装方式：

```bash
npx skills add joeseesun/qiaomu-goal-meta-skill
```

仓库内嵌了一个轻量 linter（`lint_goal_command.py`），会在输出中拦住：
- `/目标` 不可执行前缀
- `[Outcome]`、`TODO`、`待定` 等占位符
- `make sure it works` 类的模糊验证
- `edit anything` 类的无限权限
- `keep trying` 类的无限重试

这个 linter 的存在说明项目作者清楚：即使有了元技能框架，agent 仍然可能输出不可执行的内容，所以需要**双层防护**——框架约束 + 输出校验。

### 默认先推进 + 高风险才暂停

qiaomu 的内置原则很有价值：**低风险不确定性不拦用户填表**，它会做明确假设，给最佳默认方案。只有遇到凭证、账号、支付、生产数据、版权、发布时才写进暂停条件。

这个原则比"所有边界都要写"更聪明：它减少了 agent 需要处理的指令数量，同时确保高风险区域不被遗漏。

## 关联分析

qiaomu 与 **ponytail**（R536 推荐的 YAGNI 决策树项目）形成互补：

- **ponytail** 解决的是"agent 倾向于过度建设"的问题——写一个日期选择器 → 安装 npm 包 + wrapper + 样式
- **qiaomu** 解决的是"agent 的目标定义就太松"的问题——没有验证、没有边界、没有暂停条件

两者共同构成了 Loop Agent Harness 的两个维度：**目标定义层**（qiaomu）和**执行决策层**（ponytail）。

同时，qiaomu 的"先推进，遇到高风险才暂停"原则，也与 **Claude Code 的 auto mode**（Anthropic 工程博客，Mar 2026）形成有趣的呼应：两者都在探索如何在 agent 的自主性和安全性之间找到工程化平衡。

## 适用场景

**适用**：
- 开发网站、App、游戏、Chrome 插件、自动化脚本
- 修 bug、做重构、加测试、跑发布
- 把一句模糊需求变成可执行任务

**不适用**：
- 一句话翻译、简单改写
- 不需要持续执行和验证的小任务
