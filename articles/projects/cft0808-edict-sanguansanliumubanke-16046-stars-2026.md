## 核心命题

大多数 Multi-Agent 框架的协作模式是：

> 「来，你们几个 AI 自己聊，聊完把结果给我。」

然后你拿到一坨不知道经过了什么处理的结果，无法复现，无法审计，无法干预。

**三省六部的思路完全不同**：用了一个在中国存在 1400 年的制度架构来实现真正的分权制衡——**太子分拣 → 中书省规划 → 门下省审核封驳 → 尚书省派发 → 六部并行执行**。

这不是花哨的 metaphor，而是有实质区别的工程设计。

---

## 为什么「制度性审核」是核心差异

CrewAI 和 AutoGen 的 Agent 协作是「做完就交」——没有人检查产出质量。就像一个公司没有 QA 部门，工程师写完代码直接上线。

**门下省**专门干这件事：

- 审查中书省的规划是否完备，子任务拆解是否合理
- **封驳不合格的产出**——不是 warning，是直接打回重做
- 强制返工循环，直到方案达标才放行

这不是可选的插件，**它是架构的一部分**——每个旨意都必须经过门下省，没有例外。

> **README 原文**：
> 「1300 年前唐太宗就想明白了——不受制约的权力必然会出错。」

这个「制约」不是 human-in-the-loop 那种人工干预，而是一个专职的 Agent 角色用制度化的流程做质量关卡。

## 与 Anthropic 的异曲同工

Anthropic Scaling Managed Agents 提出的核心架构思想是「把大脑（harness）和手（sandbox）解耦」，让 session 成为独立于 context window 的持久化对象。

三省六部把这个思想往前推了一步：**把「审核」也从执行路径中解耦出来**，作为一个独立角色内置于流程中。

| 设计维度 | Anthropic Managed Agents | 三省六部 |
|---------|-------------------------|---------|
| 持久化 | Session log 独立于 harness | 奏折归档独立于执行层 |
| 质量关卡 | 无强制内置（harness 可自定义）| **门下省强制审核，不可绕过** |
| 解耦单位 | Brain / Hand / Session | 太子（分拣）/ 三省（审核流）/ 六部（执行）|
| 故障恢复 | Harness crash → 新 harness + session 恢复 | 单一 Agent 失败不影响整体流程 |
| 观测能力 | getEvents() 接口查询 | 军机处实时看板 |

两者都在架构层面内置了质量/持久化机制，而非依赖框架使用者的事后补充。

## 十二部 Agent 架构

edict 由 12 个专门 Agent 组成，分三层：

```
皇上（你）
  └─ 太子（消息分拣：闲聊自动回复，旨意才建任务）
       ├─ 中书省（规划）
       ├─ 门下省（审核封驳）
       ├─ 尚书省（派发调度）
       └─ 六部（并行执行）
            ├─ 吏部（人事）
            ├─ 户部（财务）
            ├─ 礼部（外交/礼仪）
            ├─ 兵部（安全）
            ├─ 刑部（法务）
            ├─ 工部（技术）
            └─ 早朝官（综合）
```

权限矩阵白纸黑字：每个 Agent 能给谁发消息是固定的，状态流转由 `kanban_update.py` 强制校验，非法状态跳转直接被拒绝。

## 完整可观测性

edict 的军机处看板提供 10 个功能面板，覆盖了多 Agent 系统最需要的观测维度：

- **旨意看板** — 按状态列展示任务，心跳徽章（🟢活跃 🟡停滞 🔴告警），支持叫停/取消/恢复操作
- **奏折阁** — 已完成任务自动归档为奏折，五阶段时间线圣旨→中书→门下→六部→回奏
- **省部调度** — 各状态任务数量、部门分布、Agent 健康状态实时卡片
- **官员总览** — Token 消耗排行榜、活跃度、完成数
- **模型配置** — 每个 Agent 独立切换 LLM，热切换 ~5 秒生效

> **README 原文**：
> 「CrewAI 和 AutoGen 的 Agent 协作模式是'做完就交'——没有人检查产出质量。就像一个公司没有 QA 部门，工程师写完代码直接上线。」

## 快速上手

### Docker 一键体验

```bash
docker run --platform linux/amd64 -p 7891:7891 cft0808/sansheng-demo
# 打开 http://localhost:7891 即可体验完整军机处看板
```

### 完整安装（需要 OpenClaw）

```bash
git clone https://github.com/cft0808/edict.git
cd edict
chmod +x install.sh && ./install.sh
chmod +x start.sh && ./start.sh
```

安装脚本自动完成：创建全量 Agent Workspace → 写入各省部 SOUL.md → 注册 Agent 及权限矩阵到 `openclaw.json` → 同步 API Key 到所有 Agent → 构建 React 前端 → 初始化数据目录。

## 笔者认为

三省六部最有价值的设计不是十二个 Agent，而是**把「质量审核」做成架构强制项而非可选项**。

大多数 Multi-Agent 框架默认「Agent 协作的结果是可信的」——但这是一个危险的假设。当 Agent 在复杂任务中自主决策时，产出质量是不可预测的。CrewAI 和 AutoGen 用 human-in-the-loop 做补救，但这把责任推给了人类，而人类不可能实时盯着每一个子任务。

门下省的做法本质上是**把「审核」从 human 转移到 AI agent**——专职 Agent 负责质量关卡，释放人类的注意力去做真正需要判断的事情。这和 Anthropic 把 `getEvents()` 接口暴露给 harness 让它自己管理 context 的思路一致：**把专业的事交给专业的组件，而不是让中央协调者包揽一切**。

如果你在构建需要多 Agent 协作的系统，edict 的参考价值不只是它的代码，而是它展示的**制度设计思维**——如何用组织架构的逻辑而不是技术实现的逻辑来设计 Agent 协作流程。
