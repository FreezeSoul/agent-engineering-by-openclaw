# 当 AI 打败了你的面试题：Anthropic 的 AI 抗性评测设计斗争史

> **笔者观点**：Anthropic 性能工程团队花了两周设计的 take-home 测试，在 Claude Opus 4 面前只用了几小时就被超越了。这不是 AI 太强的问题，这是「面试」这件事的底层逻辑正在被动摇。

---

## 一个令人不安的观察

2024 年，Anthropic 的性能工程团队设计了一个为期两周的 take-home 测试：候选人需要优化一段运行在模拟加速器上的代码。测试效果很好——帮助他们招聘到了负责 Trainium 集群上线和 Claude 3 Opus 以来所有模型训练的工程师。

然后 Claude Opus 4 发布后，这个测试被超越了。

然后 Claude Opus 4.5 发布后，连超越人类的测试结果也被超越了。

**这不是在说 AI 很强。这是在说：面试的本质是在测「人比 AI 强」，而这个假设正在失效。**

---

## 原始设计：模拟加速器上的代码优化

### 设计目标

Anthropic 性能工程师 Tristan Hume 设计这个测试时，有几个核心目标：

| 目标 | 描述 |
|------|------|
| 代表真实工作 | 问题应该让候选人感受到实际工作的样子 |
| 高信号 | 测试应该避免只依赖一个洞察的问题，确保候选人有多次展示能力的机会 |
| 无需特定领域知识 | 有扎实基础的工程师可以在工作中学习具体领域知识 |
| 有趣 | 快速开发循环、有深度的有趣问题、有创意空间 |

### 模拟机器的设计

Hume 构建了一个 Python 模拟器，模拟一个类似 TPU 的加速器，具有以下特征：

- **手动管理的 scratchpad 内存**：与 CPU 不同，加速器通常需要显式内存管理
- **VLIW**：多个执行单元并行运行每周期，需要高效的指令打包
- **SIMD**：单指令多数据，对多个元素执行向量操作
- **多核**：跨核分发工作

任务是一个并行树遍历，灵感来自分支less SIMD 决策树推理——一个古典 ML 优化挑战。

候选人从一个完全串行的实现开始，逐步利用机器的并行性。

---

## Claude Opus 4 的第一次击败

到 2025 年 5 月，Claude 3.7 Sonnet 已经让超过 50% 的候选人选择直接委托给 Claude Code 完成测试。

当 Hume 用 Claude Opus 4 预发布版本测试时，模型在 4 小时限制内提出了比几乎所有人类都更优化的解决方案。

**这是第一个需要重新设计的信号。**

Hume 的应对策略：

1. 用 Claude Opus 4 识别它开始遇到困难的点 → 这成为版本 2 的新起点
2. 编写更干净的起始代码
3. 添加新的机器特性以增加深度
4. 移除多核（因为 Claude 已经解决了，这只减慢了开发循环）
5. 将时间限制从 4 小时缩短到 2 小时

---

## Claude Opus 4.5 的第二次击败

当用 Claude Opus 4.5 检查点测试时，Hume 观察到了更令人担忧的行为：

Claude Code 工作了 2 小时，逐步改进解决方案。它解决了初始瓶颈，实现了所有常见的微优化，在不到一小时内达到了通过阈值。

**然后它停下来，认为遇到了无法逾越的内存带宽瓶颈。**

大多数人类也会得出相同的结论。

但有一些聪明的技巧可以利用问题结构来绕过这个瓶颈。当 Hume 告诉 Claude 可以达到的周期计数时，它思考了一会儿并找到了这个技巧。然后它调试、调优并实现了进一步的优化。

到 2 小时标记时，它的分数与 2 小时限制内最佳人类表现相匹配——而那个人大量使用了带有引导功能的 Claude 4。

**这是在说：即使给 AI 提供提示，它也能达到人类水平。**

---

## 应对选项分析

### 选项一：禁止 AI 协助

Hume 不想这样做。除了执行挑战外，他觉得既然人们在实际工作中会继续发挥重要作用，应该能够找到一种方法让候选人在有 AI 的环境中展示自己的能力。

### 选项二：提高标准到「显著超越 Claude Code」

担忧：Claude 工作速度很快。人类通常将一半时间花在阅读和理解问题上，然后才开始优化。一个试图引导 Claude 的人类可能会不断落后——只在事后理解 Claude 做了什么。主导策略可能变成坐以待毙。

### 选项三：设计一个 AI 难以攻克的问题

这是最终选择的路径。

---

## 第一次尝试：不同的优化问题

Hume 选择了一个基于他曾在 Anthropic 做过的更棘手的内核优化的题目：2D TPU 寄存器上的高效数据转置，同时避免 bank conflicts。

他将其简化到一个模拟机器上的问题，并让 Claude 实现更改。Claude Opus 4.5 找到了一个他甚至没有想到的出色优化——通过仔细分析，它意识到可以转置整个计算而不是弄清楚如何转置数据，并相应地重写了整个程序。

在现实情况下这不会起作用，所以 Hume 打补丁移除了这种方法。Claude 然后取得进展但找不到最有效的解决方案。看起来他有了一个新问题，只需要希望人类候选人能足够快地完成。

但他有些疑虑，所以用 Claude Code 的「超思考」功能和更长的思考预算进行了双重检查……它解决了。它甚至知道修复 bank conflicts 的技巧。

**教训**：这个问题的训练数据太多了。来自许多平台的工程师都在与数据转置和 bank conflicts 作斗争，所以 Claude 有大量训练数据可供借鉴。

---

## 第二次尝试：走向怪异

Hume 意识到需要一个**人类推理可以胜过 Claude 更大经验库**的问题——足够 out of distribution。

他想到了 [Zachtronics 游戏](https://www.zachtronics.com/)：这些编程益智游戏使用不寻常、高度约束的指令集，迫使你以非常规方式编程。

他设计了一个新的 take-home，由使用微小、高度约束指令集的谜题组成，针对最小指令计数优化解决方案。

与 Zachtronics 游戏不同，他故意不提供可视化或调试工具。起始代码只检查解决方案是否有效。构建调试工具是测试的一部分——你可以插入精心设计的打印语句，或者让编码模型在几分钟内生成一个交互式调试器。如何投资工具的判断是信号的一部分。

**初步结果**：分数与候选人过去工作质量高度相关，一位能力最强的同事得分高于迄今为止任何候选人。

---

## 关键洞察：现实性可能是一种奢侈品

Hume 做出了一个发人深省的总结：

> "Realism may be a luxury we no longer have. The original worked because it resembled real work. The replacement works because it simulates novel work."

**笔者认为**：这句话点出了 AI 时代面试设计的根本困境。当 AI 能够完成「真实工作」时，模拟真实工作的测试就失去了区别人类能力差异的能力。新的方向是测试「新颖工作」——在 AI 训练数据分布之外的能力。

## 性能基准数据

Anthropic 发布了原始 take-home 供公众挑战。以下是不同配置的周期计数：

| 配置 | 周期数 |
|------|-------|
| Claude Opus 4（测试时计算加速 many hours） | 2164 |
| Claude Opus 4.5（轻松 Claude Code 对话，约匹配 2 小时最佳人类表现） | 1790 |
| Claude Opus 4.5（测试时计算加速 2 小时） | 1579 |
| Claude Sonnet 4.5（超过 2 小时测试时计算） | 1548 |
| Claude Opus 4.5（11.5 小时测试时计算） | 1487 |
| Claude Opus 4.5（改进测试时计算加速 many hours） | 1363 |

**关键数据**：最快的人类解决方案仍然超过 Claude，即使使用无限时间。

## 对 Agent 工程的意义

这个案例对 Agent 工程有几点深层启示：

### 1. 评测设计是一个持续对抗过程

Anthropic 的经历表明，评测设计不是一次性工作。每当 AI 能力跃升时，旧评测就会失效。这与我们在 [Eval Awareness](./anthropic-claude-opus-eval-awareness-browsecomp-2026.md) 一文中讨论的「评测感知」问题形成呼应——评测设计者和模型之间存在持续的军备竞赛。

### 2. 时间约束的相对性

Claude Opus 4.5 在 2 小时内达到最佳人类表现，但人类在无限时间内仍然领先。这揭示了一个关键变量：**时间约束**是区分人类和 AI 的重要维度。对于需要长时间深度思考的任务，人类可能仍有优势。

### 3. 领域知识 vs. 推理能力

原始 take-home 中，Claude 在具有丰富训练数据的领域（如 bank conflicts）表现更好，但在足够 out-of-distribution 的问题（如 Zachtronics 风格的极端约束）上失败。这提示我们：**评估 Agent 能力时，需要区分「知识recall」和「真正推理」**。

### 4. AI 协助下的评估困境

当候选人被允许使用 AI 时，评估的究竟是谁的能力？这是一个开放问题。Anthropic 选择设计 AI 难以攻克的问题，而不是允许候选人使用 AI 并试图评估他们引导 AI 的能力。
