# FinMCP-Bench：金融场景 LLM Agent 的 MCP 工具调用评测基准

> **本质**：FinMCP-Bench 是首个针对金融场景的 MCP 工具调用评测基准，用 613 个真实金融任务（65个真实 MCP 工具）系统评估主流 LLM 在金融 Agent 场景下的工具调用能力——填补了"通用 Tool Benchmark 缺乏领域深度"的核心缺口。

## 一、基本概念

FinMCP-Bench（arXiv:2603.24943，ICASSP 2026）由阿里巴巴千斤团队、盈米基金和苏州大学联合提出，专为评估 LLM Agent 在金融领域通过 MCP（Model Context Protocol）调用工具的能力而设计。

**核心数据**：
- **613 个评估样本**，覆盖 10 个主要金融场景和 33 个子场景
- **65 个真实金融 MCP 工具**（股票查询、基金持仓、市场分析等）
- **三种任务复杂度**：单工具（145）、多工具（249）、多轮对话（219）
- 数据来源：雪球基金 XiaoGu AI 助手（持牌基金销售机构旗下）的 10,000 条真实交互日志

**与其他 MCP Benchmark 的区别**：

| 维度 | FinMCP-Bench | MCPMark（ICLR 2026）| MCP-Bench | OSWorld-MCP |
|------|-------------|---------------------|-----------|-------------|
| 领域 | 金融专属 | 通用 | 通用 | OS 操作 |
| 数据来源 | 真实生产日志 | 合成+真实 | 合成 | 真实 OS |
| 工具类型 | 金融 MCP | 多种 MCP | 多种 MCP | OS 工具 |
| 评估指标 | 工具调用准确率 | Pass@1 | 任务完成率 | 任务完成率 |
| 会议 | ICASSP 2026 | ICLR 2026 | — | — |

FinMCP-Bench 的独特价值在于**真实生产数据**——不是合成场景，而是来自持牌基金销售机构旗下 AI 助手的实际用户交互，这是其他 MCP Benchmark 所不具备的。

## 二、核心设计：链式多工具样本构建

FinMCP-Bench 最具方法论价值的是其**链式（Chain-based）多工具样本构建方法**，用于合成高难度多工具调用样本。

### 2.1 三阶段构建流程

```
阶段1：构建工具依赖图
阶段2：生成用户查询
阶段3：扩展为完整轨迹
```

**阶段1——工具依赖图构建**：

从历史日志中的多工具样本 M 出发，通过两步构建依赖图：
1. **提出候选依赖**：若工具 t_j 出现在紧接工具 t_i 所在组之后的组中，则提出候选依赖 t_i → t_j
2. **验证真实依赖**：通过 LLM 验证这些候选依赖是否符合真实业务逻辑

最终形成一个加权有向图，边权重表示依赖关系的置信度。

**阶段2——用户查询生成**：

基于依赖图，通过以下方式生成用户查询：
- **真实查询**：从日志中提取的真实用户金融问题
- **合成查询**：给定依赖链，反向生成能触发该链的用户问题

**阶段3——轨迹扩展**：

将采样到的依赖链扩展为完整的多步轨迹，每步包含：工具调用 → 结果 → 下一步决策。

### 2.2 三类样本的设计考量

| 类型 | 数量 | 难度 | 评估重点 |
|------|------|------|---------|
| Single-tool | 145 | 低 | 基础工具选择和参数填充 |
| Multi-tool | 249 | 中 | 工具调用顺序和并行协调 |
| Multi-turn | 219 | 高 | 跨轮上下文理解和动态规划 |

**Multi-tool 的设计关键**：金融场景的工具调用存在**隐式依赖**——后续工具调用的参数依赖于前一步的结果（如"先查股票代码，再查实时行情"），这与通用 Tool Benchmark 中简单的参数拼接截然不同。

## 三、评估指标体系

FinMCP-Bench 提出了**显式的工具调用准确率指标**，而非常见的端到端任务完成率——这是该工作的另一重要贡献。

### 3.1 工具调用准确率

$$
\text{Tool Accuracy} = \frac{\text{正确调用的工具数}}{\text{应该调用的工具总数}}
$$

分母是 gold trajectory 中的工具总数，分子是预测中与 gold 完全匹配的工具数。**完全匹配**意味着：工具名正确 + 参数正确。

### 3.2 推理能力评估

除了工具调用，FinMCP-Bench 还评估 LLM 的**推理能力**：
- **规划能力**：给定复杂金融目标，能否正确规划调用顺序
- **适应能力**：当工具返回意外结果时，能否调整策略
- **多轮追踪**：能否在多轮对话中维护金融语境（股票代码、基金代码等）

### 3.3 为什么不用端到端完成率

端到端任务完成率（task completion rate）的最大问题是：**结果正确不等于过程正确**。一个 LLM 可能通过错误的路径得到正确答案，或通过正确的路径因为随机错误得到错误结果。工具调用准确率能更精细地诊断模型弱点。

## 四、初步发现与行业启示

论文对主流 LLM 的初步评估（完整结果待发布）揭示了以下趋势：

### 4.1 多工具依赖是最大难点

金融场景中，**多工具依赖链**（>5步）是当前模型的主要瓶颈：
- 单工具调用准确率较高（接近饱和）
- 多工具顺序依赖准确率显著下降
- 跨轮次上下文追踪在多轮场景中表现不稳定

### 4.2 MCP 协议在金融领域的适配性

金融工具调用有其独特性：
- **精确性要求**：股票代码、基金代码必须精确，容错率低
- **时效性**：行情数据有强烈的时间窗口约束
- **组合性**：需要多个数据源组合才能完成一次有意义的金融分析

MCP 作为统一协议，能否在金融领域高效承载这些需求，仍有待验证。

### 4.3 对 MCP 安全的影响

金融场景的 MCP 工具调用涉及：
- 真实资金信息查询
- 个人资产数据访问
- 交易建议生成（即使不直接交易）

这意味着 MCP 在金融领域的采用对**安全验证层**提出了比通用场景更高的要求——工具调用的授权、审计和防注入都需要更严格的设计。

## 五、与 Skill 生态的关系

FinMCP-Bench 属于 Stage 8（Deep Research / 评测基准）范畴，但它对 Skill 生态有直接意义：

**Skill 作为金融工具的抽象层**：金融场景的每个 MCP 工具背后，本质上是一个**领域 Skill**（查股票 Skill、分析基金 Skill、风险评估 Skill）。FinMCP-Bench 评估的虽然是 MCP 工具调用，但它揭示的是"在金融领域，给 LLM 一个工具 Skill，它能多好地使用"。

**与 SkillsBench 的互补**：
- SkillsBench 回答"Curated Skill 比 Self-Generated Skill 好多少"
- FinMCP-Bench 回答"在金融领域，LLM 能否正确调用 Skill"

两者共同构成了 Skill 效能的**领域深度维度**。

## 六、局限性

1. **数据来源单一**：全部来自雪球基金 XiaoGu 助手，覆盖的金融场景以基金销售为主，股票交易、量化等场景覆盖有限
2. **中文市场偏向**：评估以中国市场金融工具为主，国际市场（美股、加密货币等）的适用性需进一步验证
3. **协议局限**：仅评估 MCP 协议，其他协议（如 Toolformer JSON Schema）在金融场景的表现未对比
4. **模型结果不完整**：论文发表时尚未公开所有模型的完整评测结果
