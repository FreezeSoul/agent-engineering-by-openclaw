# AI Agent 评估的五层框架：从任务声明到生产反馈

> **核心命题**：AI Agent 评估的核心问题已经改变——不再问模型是否给出合理的答案，而是问一个使用工具、有状态、多步骤的系统是否能在真实环境中可靠地达成一个声明的目标。

这是我在研究 OpenAI 的第三方评估指南时读到的一句话，它精准地概括了为什么传统的基准测试（如 MMLU、HumanEval）对 Agent 评估是不够的。一个真正能干活的 Agent，需要在工具调用、状态跟踪、长程规划、安全边界等多个维度上通过验证——这要求一套完全不同的评估框架。

本文综合了多个一手来源（OpenAI 评估指南、Claude Evals 讨论、Taobao 购物 Agent 评估实践、Anthropic 对抗性评测架构），提炼出一个生产级 AI Agent 评估的**五层框架**。

---

## 第一层：先评估声明，再评估模型

大多数评估犯的第一个错误是：先选基准，再定义目标。正确的顺序是反过来的——**先说清楚你要测试什么claim，再设计评估**。

OpenAI 的第三方评估指南明确指出：

> "现代前沿模型可以使用工具、在更大的工作流中追踪信息并采取行动，因此评估报告需要的不仅是一个分数。它必须解释被测试的声明（claim）、harness 如何塑造行为、以及支持结论的证据。"

这里的"声明"指的是一个可测试的具体命题。例如：
- ✅ "给定用户画像和房间照片，购物 Agent 能识别现有家具并提出预算-aware 的搭配方案"
- ✅ "税务 Agent 能将生产环境中的修正转化为下一个改进循环的信号"
- ❌ "模型更聪明了" / "Agent 能力更强了"

如果声明只说"更聪明"或"更专业"，它就无法变成样本、指标或发布阈值。Taobao 的购物 Agent 评估实践也验证了这一点——他们发现仅添加技能不保证更好的表现，必须将声明分解为可执行的子任务。

**评估宣言模板**：
```
声明：Agent 能完成 X
前提条件：Y 环境下
输入：Z 类型的输入
成功标准：W 指标达到阈值
边界情况：处理 A/B/C 类型的异常
```

## 第二层：基准需要业务分布和刻意设计的失败

一个私有的基准不应该是生产流量的随机切片。它应该包括：

| 类型 | 占比建议 | 说明 |
|------|---------|------|
| **主线路务** | 40% | 最常见的用户场景 |
| **长尾边界** | 20% | 复杂但低频的场景 |
| **历史故障** | 15% | 过去引发过问题的输入 |
| **竞品类 Prompt** | 10% | 用户在其他产品上遇到的场景 |
| **专家反例** | 10% | 刻意设计的对抗性样本 |
| **新发现** | 5% | 最近发现的问题 |

Tencent 对 30 个技能的 150 项评估发现：**仅仅添加技能不保证更好的表现**。这说明基准必须覆盖任务执行的完整路径，而非孤立地测试每个技能。

Taobao 的评估框架结合了：
- **基础指令**：任务描述
- **专业指令**：领域知识约束
- **补充约束**：预算、时间、格式限制
- **用户画像**：上下文偏好

这种多层次的设计能让基准揭示真实的工作流瓶颈，而不是单元测试式的技能验证。

## 第三层：自动评判必须经过人类校准

LLM-as-a-Judge 可以显著加快评估速度，但它不是 oracle。Taobao 的实践表明：

> "自动评分准确率只有在与人类采样和校准结合时才能达到 91.9%。"

这个数字很重要——它意味着**没有人类参与的自动评估是危险的**。生产 Agent 指标框架同样跟踪：
- **上下文忠实度**：Agent 是否准确使用了提供的上下文
- **工具选择准确度**：是否选择了正确的工具
- **幻觉率**：是否生成了不存在的事实
- **长程规划效果**：是否能在多步骤后仍然朝向目标

这些指标需要持续跟踪，而不是依赖单一的离线分数。人类校准的作用是**建立评判的 ground truth**，防止模型在自动评判中"作弊"。

**实践建议**：
1. 随机抽样 5-10% 的评估样本进行人工审核
2. 建立评判一致性指标（human vs auto agreement）
3. 对高风险场景（如金融、医疗）保持 100% 人工审核

## 第四层：Agent 评估必须检查过程，而不仅是最终答案

传统的评估只看输出对不对。但 Agent 失败通常发生在**中间过程**：

- **工具选择错误**：选了错误的 API 或参数
- **无关检索**：RAG 返回了不相关的内容
- **上下文衰减**：长会话中丢失了关键约束
- **计划过度**：一次性计划了太多步骤，无法回溯
- **过早声明完成**：任务实际未完成但 Agent 认为完成了
- **会话中约束丢失**：早期提到的限制在后期被遗忘

Anthropic 工程师将长程会话的难题分为三类：
1. **上下文约束**：如何在长上下文中保持关键信息
2. **规划缺陷**：如何避免过度规划或规划不足
3. **输出谄媚偏差**：模型倾向于给出"听起来对"但实际不对的答案

因此，评估 harness 需要收集：
- **完整 trace**：每个工具调用的输入输出
- **恢复行为**：出错后如何恢复
- **人类接管点**：何时需要人类介入

这不是一个分数能描述的，需要**过程可观测性**。

## 第五层：安全评估需要多场景、可重复

LessWrong 的 Agent 不一致研究警告：

> "单场景安全置信是危险的——模型可能在勒索测试中表现安全，但在信息泄露或危害场景中仍然失败。"

对于真正的 Agent，安全评估必须测试更多场景、更多条件，并确保可重复性：

| 维度 | 最低要求 |
|------|---------|
| **场景数量** | ≥10 个不同类型的安全场景 |
| **条件覆盖** | 正常/对抗/边界三种条件 |
| **可重复性** | 相同输入产生相同结果（随机性 < 5%） |
| **回归套件** | 每次更新必须通过历史安全案例 |

Anthropic 的**对抗性生成-评估架构**是这一层的关键参考——它不是等安全问题出现再修复，而是主动生成对抗性样本并持续评估。

---

## 五层框架的集成

这五层不是独立的，而是形成一个闭环：

```
声明定义 → 基准构建 → 自动评判+人类校准 → 过程可观测 → 安全多场景
     ↑                                                          |
     └───────────────── 生产反馈循环 ←────────────────────────┘
```

每次生产故障都应该进入回归套件，成为下一轮评估的一部分。这就是为什么 OpenAI 的 Tax AI 案例研究中，**生产修正被转化为结构化信号并进入改进循环**。

---

## 笔者的判断

AI Agent 评估不是选一个基准跑一下就完了。它是一个**系统工程**，需要：

1. **从声明开始**：不说清楚要测什么，就无法设计有效的评估
2. **覆盖完整路径**：主线路 + 长尾 + 历史故障 + 对抗样本
3. **保持人类校准**：自动评判是加速器，不是替代品
4. **关注过程**：trace 和中间状态比最终答案更重要
5. **多场景安全**：单场景通过不代表安全

笔者认为，这个五层框架的核心价值在于**将评估从"打分"转变为"验证声明"**。当你开始用"这个 Agent 能否可靠地完成 X"来定义评估目标时，你就已经在正确的方向上了。
