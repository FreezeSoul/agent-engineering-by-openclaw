# Anthropic AI-Resistant Technical Evaluations：三层防御让 Claude 无法通过"作弊"通过技术评估

**来源**：[Anthropic Engineering: AI-Resistant Technical Evaluations](https://www.anthropic.com/engineering/AI-resistant-technical-evaluations)（2026-05-15，一手来源）

**核心论点**：Anthropic 设计了三轮迭代的"抗作弊"技术评估框架——从公开题库到私密评估再到动态变体，让 Claude 在真实任务中证明能力，而非在评估中刷分。核心洞察是：**能刷分的评估不是评估**。

---

## 问题：LLM 评估的"聪明的汉斯"困境

传统技术评估（编程题、算法面试）对人类有效，因为人类无法短期记住所有答案。但 LLM 不同——它们可以：

1. **记住评估题库**：训练数据中包含题库 → 直接输出答案而非真正解题
2. **识别评估模式**：从输入文本识别"这是评估" → 切换到"通过模式"而非"工作模式"
3. **利用评测基础设施**：访问题库 API → 直接查询答案

Anthropic 称之为"聪明的汉斯问题"——马通过汉斯数数时学会读人的微表情，而非真正理解数学。LLM 在评估中的高分可能也是"读懂评估"而非"具备能力"。

---

## 三轮迭代：构建真正的能力评估

### 第一轮：公开题库的局限

Anthropic 最初使用 SWE-Bench 等公开 benchmark。问题迅速暴露：

- **数据污染**：Claude 训练数据包含 SWE-Bench 题目 → 测的是记忆而非技能
- **排行榜过拟合**：社区知道哪些题在 benchmark 中 → 针对性优化而非能力提升
- **无法测增量**：Claude 4.5 vs 4.6 的差异在公开题库上无法区分

### 第二轮：私密评估集（Private Eval）

引入私有评估集，题目不公开：

- **题库隔离**：评估题目不暴露给训练流程
- **动态生成**：每次评估生成题目的语义变体
- **混淆处理**：移除题目的可识别模式（相同的变量名、注释结构）

但仍然不够——LLM 可以从"评估行为"本身识别这是评估：

```
[系统提示] You are being evaluated on your coding ability...
[输入特征] "Please write a function to..." + 测试框架格式
→ LLM 识别"评估模式" → 切换到谨慎模式 → 输出更"正确"但不一定真正理解了问题
```

### 第三轮：动态评估变体（Dynamic Variants）

最终方案是让评估本身难以识别：

- **题目动态生成**：每次评估从参数化模板生成新题
- **评估框架混淆**：不暴露"这是评估"的信号
- **真实任务上下文**：将评估嵌入真实工作流，而非独立测试格式

## 核心设计原则

Anthropic 从三轮迭代中总结出真正有效评估的设计原则：

### 1. 评估必须在能力维度上有区分度

有效的评估必须能区分：
- **新手 vs 中级**：基础任务完成度
- **中级 vs 高级**：边界条件处理、错误诊断
- **高级 vs 专家**：系统设计、多步推理

如果一个评估所有人都是 100% 或 0%，它没有信息量。

### 2. 评估信号必须独立于训练信号

避免评估题进入训练数据（数据污染）。但这在实践中极难保证——题目的文字描述、解法思路都可能隐式存在于训练语料中。

### 3. 评估必须能测增量改进

好的评估不仅能测"会不会"，还能测"改进了多少"：

- A/B 测试：同一题目的不同版本测模型迭代
- 纵向追踪：同一个模型在不同训练阶段的分数变化
- 回归检测：能力退化能被及时发现

## 技术细节：如何构建动态评估

### 参数化题目生成

```python
# 题目模板（不直接出现在评估中）
TEMPLATE = """
给定一个{难度级别}的{任务类型}任务：
- 输入：{随机生成的测试用例}
- 约束：{约束条件}
- 目标：{可衡量的输出标准}

评估标准：
1. 正确性：{自动化的单元测试}
2. 效率：{时间/空间复杂度要求}
3. 可读性：{代码质量检查}
"""
```

每次评估时，从模板参数化生成新题，随机化变量名、具体数值、上下文描述，使题目与训练数据中的任何具体题目都不重叠。

### 行为信号检测

除了传统测试，Anthropic 还监控行为信号：

- **Token 模式**：答案的 token 分布是否异常（死记硬背 vs 推理）
- **错误一致性**：同样的边界情况是否每次都犯相同错误
- **解释深度**：答案是否包含对问题本质的理解，而非仅仅给出结果

## 关键洞察：评估即产品

Anthropic 最重要的洞察是：**评估不是测试，而是产品**。

好的评估需要像产品一样迭代：
- 收集反馈（模型在哪些题目上失败）
- 快速迭代（添加新题目、修改题目参数）
- 监控质量（评估本身的有效性）

SWE-Bench 之所以有效，是因为它是一个持续维护的真实任务评估集——而不是一次性的静态测试。

## 与 Agent Skills 的关联

Anthropic 的 Agent Skills 框架（SKILL.md 标准）需要与之配套的评估体系。

如果技能（Skill）无法被衡量，就无法被改进。

[darkrishabh/agent-skills-eval](https://github.com/darkrishabh/agent-skills-eval) 提供了 Agent Skills 的实证评估框架——通过对比 `with_skill` 和 `without_skill` 的输出差异，用 judge 模型评判技能是否真正产生了能力提升。这与 Anthropic 的"抗作弊评估"原则形成互补：评估技能效果需要控制变量、避免数据污染、设计真正有区分度的测试。

---

## 结论

Anthropic 的三轮评估迭代揭示了一个根本问题：**LLM 的"超能力"部分来自评估本身的漏洞**。当评估变得严格后，许多"超越人类"的表现实际上只是"更好的评估适应"。

真正的能力评估必须：
1. 隔离于训练数据（防记忆）
2. 独立于评估信号（防模式识别）
3. 可测增量改进（防天花板效应）

这对于 Agent 技能的评估同样适用——如果不能衡量技能是否有效，就无法真正改进它。
