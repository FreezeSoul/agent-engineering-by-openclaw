# Agent Evaluation Readiness Checklist：构建生产级 Agent 评测的系统方法论

> **核心问题**：大多数团队在构建 Agent 评测时，先建基础设施再想评测目标，导致评测跑了很多却没有测量价值。本文拆解 LangChain 的 Agent Evaluation Readiness Checklist，探讨如何从"测量什么"出发，反推评测基础设施的构建路径。

---

## 为什么现有评测方法论不够用

传统的软件测试是确定性的——给定输入 A，期望输出 B，测试逻辑清晰。但 Agent 的输出是非确定性的：同一个任务，模型可能走不同路径得到等效结果，或者在边界情况下做出不可预期的行为。

这导致两种常见的失败模式：

1. **基础设施先行陷阱**：先搭建好评测 pipeline，再填入"随便测测"的任务，得到一堆数字但无法指导决策
2. **评测目标缺失**：没有明确"成功是什么"，两个团队成员对同一输出是否"正确"都无法达成一致

LangChain 的 Agent Evaluation Readiness Checklist 的核心主张是：**在触碰任何评测代码之前，先回答三个问题：我们测什么、成功的标准是什么、谁来维护这个标准**。本文拆解这个 checklist 的结构化执行路径。

## 第一阶段：构建评测前的检查清单

### 1.1 人工审查 20-50 条真实 Agent Trace 是最高性价比的投资

在搭建任何自动化系统之前，用 30 分钟通读真实 Agent 执行记录。这个动作的价值超过任何评测框架：

- 从 LangSmith 的 Traces 和 Annotation Queue 可以看到真实用户的交互路径
- 手动审查揭示的是系统性失败模式，而非单个错误
- 在基础设施不成熟时，人工审查的准确度远高于自动化评测

**具体操作**：随机抽取 20-50 条完整 trace，按"意图理解→工具选择→执行结果→最终响应"四阶段标记异常点。这个过程通常会发现 5-8 类重复出现的失败模式，这些就是后续评测应该覆盖的核心场景。

### 1.2 区分 Capability Eval 和 Regression Eval

这是两个目标截然不同的评测类型：

**Capability Eval**（能力评测）回答"Agent 能做什么"：
- 初始阶段通过率低（10-30%）
- 目标是持续提升，推动 Agent 在更难任务上成功
- 失败是预期的，用于发现改进方向

**Regression Eval**（回归评测）回答"Agent 是否退化"：
- 期望通过率接近 100%
- 目标是防止已解决问题的复现
- 任何失败都是需要立即修复的信号

两者的数据集不能混用。Capability Eval 的失败样例是未来改进的目标，但不应该作为 Regression Eval 的通过标准。如果不分离这两种评测，团队会陷入"不断降低成功标准以提升通过率"的反模式。

### 1.3 失败归因的 60-80% 法则

LangChain 引用了一个关键数据：**评测工作的 60-80% 应该花在失败归因上**。这与直觉相反——大多数团队以为评测的核心是"跑测试"，但实际上"分析失败原因"才是价值所在。

失败归因的标准流程：
1. **收集**：从生产或测试环境收集代表性失败样本
2. **开放编码**：不预设分类，由领域专家逐条标记问题类型
3. **归类**：将问题归纳为失败分类体系（Prompt 问题、工具设计问题、模型局限、工具失败、数据缺口等）
4. **迭代**：持续审查直到不再发现新的失败类型

归因的结果直接指导修复方向：
- **Prompt 问题**：指令不清晰导致 Agent 误解 → 修复 Prompt
- **工具设计问题**：工具接口设计让 Agent 容易犯错 → 重新设计参数或边界
- **模型局限**：指令清晰但 LLM 泛化能力不足 → 增加示例或更换模型架构

如果无法清晰归因，说明需要更多失败样本的收集，此时不应搭建自动化评测系统。

## 第二阶段：选择正确的评测层级

### 2.1 三个评测层级：Run / Trace / Thread

Agent 评测分为三个层级，每层测试不同的行为维度：

**Run-level（单步评测）** 回答"Agent 是否选择了正确的工具"：
- 最容易自动化（确定性检查）
- 需要 Agent 架构稳定（工具定义不能频繁变化）
- 适用于工具选择逻辑是核心的场景

**Trace-level（全轨迹评测）** 回答"Agent 是否正确完成了任务"：
- 评估最终响应质量 + 执行路径合理性
- 这是大多数团队应该**首先启用**的层级
- 评估三个维度：最终响应是否正确、路径是否合理、状态变更是否正确

**Thread-level（多轮会话评测）** 回答"Agent 是否保持上下文连贯性"：
- 最难实现，依赖真实对话数据
- 建议使用 N-1 测试法：用生产环境的真实前 N-1 轮对话作为上下文，让 Agent 只生成第 N 轮，评估生成质量，避免合成多轮数据的累积误差问题

**建议的启动顺序**：Trace-level → Run-level → Thread-level（按需逐步引入）

### 2.2 状态变更评估：被忽视的关键维度

对于执行型 Agent（不只是生成文本的 Agent），状态变更评估是必须覆盖的维度：

- Agent 说"会议已安排"不代表日历事件真实创建
- Agent 说"代码已写入"不代表文件内容正确
- Agent 说"数据库已更新"不代表行的值符合预期

验证状态变更的标准方法：
- 调用了 API？查 API 响应
- 写入了文件？读回文件内容
- 更新了数据库？执行 SELECT 查询

只评估最终文本输出会掩盖"Agent 说了正确的事但做错了"这类最危险的生产故障。

## 第三阶段：构建评测数据集

### 3.1 任务无歧义性是评测有效性的前提

评测任务必须满足两个条件：任务描述无歧义、且有参考解决方案证明任务可完成。

**歧义示例 vs 清晰示例**：
- ❌ 歧义："总结这篇文档"
- ✅ 清晰："从这篇会议记录中提取 3 个主要行动项，每个格式为：[执行人] - [具体任务] - [截止日期]"

如果两个专家无法对同一输出的正确性达成一致，任务本身就定义不清，需要重新设计任务而非责怪 Agent。

### 3.2 负面案例与正面案例同等重要

如果只测试"Agent 应该做什么"，优化方向会偏向"过度泛化"。必须同时包含**故意设计为假的场景**，验证 Agent 的拒绝能力。

负面案例设计思路：
- 应该拒绝的场景：请求敏感操作、超出能力范围的任务、模糊到无法完成的指令
- 每次添加新的 Capability Eval 时，优先设计对应的负面案例

### 3.3 数据集来源的三层结构

**第一层（冷启动）**：手动创建 20-50 个覆盖关键维度的种子任务。质量优先于数量，20 个经过人工验证的可靠任务胜过 200 个未验证的合成任务。

**第二层（规模扩展）**：从外部基准测试（Terminal Bench、BFCL）中**精选**适合自己 Agent 能力范围的子集，而非跑完整基准测试。完整基准是评测框架，不是评测任务。

**第三层（持续输入）**：建立 dogfooding 流程——团队日常使用 Agent，将每一次失误转化为新的评测任务。这是生产级评测数据集最可靠的来源。

## 第四阶段：Grader 设计

### 4.1 四大 Grader 类型与适用场景

| Grader 类型 | 适用场景 | 注意事项 |
|-------------|---------|---------|
| **Code-based** | 确定性检查：工具调用验证、输出格式、执行结果 | 可能因有效但非预期格式而误报 |
| **LLM-as-judge** | 主观评估：质量评分、开放任务判断 | 需要与人类偏好对齐的校准过程 |
| **Human** | 边界案例：歧义输出、关键决策审查 | 成本高、速度慢 |
| **Pairwise** | 版本对比：模型 A vs 模型 B 哪个更好 | 相对评估，不产生绝对分数 |

**核心原则**：能用确定性验证就不用 LLM-as-judge。LLM-as-judge 对客观任务（如"输出是否为有效的 JSON"）的判断不如正则表达式可靠，且存在不一致性问题。

### 4.2 Guardrail vs Evaluator：不要混淆两个概念

**Guardrail（防护栏）**：
- 执行中触发（在用户看到输出前）
- 毫秒级延迟
- 目的：阻止危险或格式错误的输出

**Evaluator（评测器）**：
- 异步触发（在生成后）
- 秒到分钟级延迟
- 目的：测量质量、发现回归

混淆两者的后果：
- 把安全检查放在异步评测路径中 → 危险输出先到达用户
- 把质量评测放在执行路径中 → 每次响应延迟数秒

### 4.3 Grader 校准：20+ 标注样本起步

LLM-as-judge 的校准流程：
1. 准备 20+ 有人工标注的样本（输入+预期评分）
2. 用 LangSmith 的 Align Evaluator 工具对比 judge 输出与人工标注的差异
3. 迭代至校准收敛后扩展至 ~100 样本用于生产级置信度

**关键要求**：Judge 的输出必须包含推理过程（reasoning）。这有两个价值：
- 推理链本身帮助审核员判断评分是否合理
- Anthropic 的研究表明，推理链输出能提升评测准确性

**持续校准**：Judge 会随时间漂移，没有单一 judge 在所有基准上表现一致。建议定期用同批标注数据重新校准，并在产品迭代后更新 few-shot 示例。

## 第五阶段：运行与迭代

### 5.1 三种评测模式与使用时机

| 模式 | 时机 | 用途 |
|------|------|------|
| **Offline** | 部署前 | 用精选数据集测试变更，迭代改进 |
| **Online** | 生产环境 | 持续监控真实流量中的表现 |
| **Ad-hoc** | 探索性分析 | 发现未预期的问题（用 LangSmith Insights）|

大多数团队在 Offline 上投入过度，在 Online 和 Ad-hoc 上投入不足。Offline 评测告诉你"改了什么"，但 Online 评测告诉你"生产环境实际发生了什么"。

### 5.2 非确定性 Agent 的评测策略

模型输出在多次运行间有变化。对非确定性 Agent：
- 使用多个重复试验（multiple repetitions）测量
- 计算置信区间而非依赖单次结果
- 考虑 pass@k（k 次中至少一次成功）或 pass^k（k 次全部成功）指标

同时追踪效率指标（步数、工具调用数、延迟、成本），一个准确率 95% 但延迟 10 倍的 Agent 未必是改进。

### 5.3 任务失败 vs 评测失败

当一个测试失败时，需要判断：
- **任务失败**：Agent 确实没做对
- **评测失败**：Grader 判断错误（误报或漏报）

**工程建议**：每次评测实验附加元数据标签（评测类别、测量维度），实验结果按标签分组查看。如果某一类任务持续出现"人工审查认为正确但评测认为失败"，说明该类别的 Grader 需要重新校准。

## 评测成熟度演进路径

Agent 评测不是一次性项目，而是随 Agent 能力增长的持续工程投资。以下是演进路径：

**阶段 1（0-1 条迹）**：手动审查 Trace，人工定义成功标准，建立第一个 regression eval 基线。

**阶段 2（1-10 条迹）**：分离 capability/regression eval，建立数据集 flywheel，配置 LLM-as-judge 校准流程。

**阶段 3（10+ 条迹）**：多层级评测覆盖（Run/Trace/Thread），Online eval 接入，Tag 系统和效率指标追踪，Grader drift 监控。

---

## 核心判断：评测的价值在于归因质量，而非覆盖率

本文的核心结论不是"如何建立更完整的评测"，而是**评测工作的价值与失败归因的质量直接相关**。60-80% 的评测工作量应该花在归因分析和 Grader 校准上，而非追求评测任务数量的增长。

一个覆盖 5 类核心场景、每次失败都能追踪到具体根因的评测体系，远比一个覆盖 500 个任务但每次失败都无法解释的评测体系更有价值。
