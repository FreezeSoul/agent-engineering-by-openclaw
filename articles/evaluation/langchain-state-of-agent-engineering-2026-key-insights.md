# LangChain Agent 工程现状报告 2026：5 个关键工程洞察

> **核心论点**：这份覆盖 1340 名专业人员的调查揭示了 Agent 工程正在从「概念验证」走向「生产可靠」——但品质、上下文治理、安全的分层挑战才刚刚开始。

---

## 调查背景

LangChain 于 2025 年 11-12 月调研了 1340 名专业人员，涵盖工程师、产品经理、企业领导。63% 来自科技行业，10% 金融，6% 医疗，9% 来自 10k+ 规模企业。

调查原文：https://www.langchain.com/state-of-agent-engineering

---

## 洞察 1：生产落地正在加速，但大规模企业领先

**关键数据**：57.3% 的受访者已有 Agent 在生产环境运行，较去年（51%）显著增长。另有 30.4% 正在开发中。

规模差异明显：
- **10k+ 规模企业**：67% 有生产 Agent，24% 在开发
- **< 100 人小团队**：50% 有生产 Agent，36% 在开发

**笔者认为**：大企业在平台团队、安全和可靠性基础设施上的投入正在拉开差距。10k+ 企业的 67% vs 小团队的 50%，差距不是技术问题，而是**投资意愿和组织能力的问题**。中小团队的方向不是「追上」，而是「找到自己的落地场景」。

---

## 洞察 2：品质是生产的头号杀手，成本不再是问题

**关键数据**：32% 将品质列为主要障碍（准确率、相关性、一致性、品牌调性），其次是延迟（20%），成本关注度**显著下降**。

在 2k+ 规模企业中，安全成为第二关注点（24.9%），超过延迟。这在 10k+ 企业中更明显：幻觉和一致性是最突出的品质问题，同时「上下文工程」和「大规模上下文管理」成为高频词。

**笔者认为**：成本不再是核心问题，这个转变非常重要——这意味着模型价格已经降到了企业可以接受的阈值，企业注意力从「多少钱」转向了「能不能用」。但品质问题说明**Harness 层（评估器、上下文管理、权限控制）还没有被工程化到位**。这是 2026 年的工程重点。

---

## 洞察 3：可观测性是标配，但评估仍在追赶

**关键数据**：89% 的组织已为 Agent 实现了某种形式的可观测性，62% 实现了详细追踪（可检查每个 Agent 步骤和工具调用）。

但评估（Evaluation）仍然落后：只有 52.4% 运行离线评估，37.3% 运行在线评估。**最大的缺口在于「没有评估」的比例从整体受访者的 29.5% 降到已有生产 Agent 组的 22.8%**——说明正在迎头赶上，但整体覆盖率仍不足。

评估方法上：
- **人类审查（59.8%）** > **LLM-as-Judge（53.3%）** > ROUGE/BLEU（有限）
- 有评估的组织中，近 1/4 同时使用离线 + 在线评估

**笔者认为**：可观测性先行、评估滞后的模式是合理的——先知道发生了什么，再判断发生的事对不对。但 52.4% 的离线评估覆盖率说明**评估基础设施还是不够成熟**。评估是 Agent 改进闭环的核心，评估缺位意味着无法系统性地改进 Agent。

---

## 洞察 4：多模型是常态，微调不是标准实践

**关键数据**：
- **68% 使用 OpenAI GPT 模型**
- **76% 在生产或开发中同时使用多个模型**
- **33% 部署了自己的模型**（开源模型，数据主权需求，监管约束）
- **57% 不进行微调**，依赖基础模型 + Prompt 工程 + RAG

**笔者认为**：多模型路由已成主流——按复杂度、成本、延迟选择模型，而不是押注单一供应商。但微调不流行的原因值得深思：**数据收集、标注、训练基础设施和维护的成本太高**。在大多数场景下，Prompt 工程 + RAG 的组合已经足够好。但对于高价值、高影响的专用场景，微调仍有价值——只是门槛高，不适合大多数团队。

---

## 洞察 5：编码 Agent 主导日常工作，上下文工程成为隐形瓶颈

调查的自由填写回答揭示了一个现实：
1. **编码 Agent 主导日常**：Claude Code、Cursor、Copilot、Windsurf 是被提及最多的工具，用于代码生成、调试、测试创建、大型代码库导航
2. **研究 Agent 第二**：ChatGPT、Claude、Gemini、Perplexity 用于领域探索、长文档摘要、跨源信息综合
3. **自定义 Agent 正在出现**：基于 LangChain/LangGraph 构建的 QA、内部知识库搜索、SQL/text-to-SQL、需求规划、客户支持、工作流自动化

对于 10k+ 规模企业，**内部生产力**（26.8%）超过客户服（24.7%）成为首要场景，说明大企业倾向于先内部效率化，再面向客户。

**笔者认为**：编码 Agent 的主导地位验证了一个判断——**Agent 工程最成熟的落地场景是软件开发**，而不是客服或研究。这与 SWE-bench、Terminal-Bench 等评估基准的流行形成正循环。

---

## 笔者的核心判断

这份报告最重要的信号不是数据本身，而是数据透露的**工程成熟度分层**：

| 层级 | 状态 | 代表性问题 |
|------|------|-----------|
| **可观测性** | 基本解决（89%） | 已有成熟工具和实践 |
| **评估** | 正在追赶（52%） | 缺乏系统化的质量量化体系 |
| **上下文工程** | 刚刚出现（被高频提及） | Agent 行为不一致的根因之一 |
| **安全沙箱** | 企业级需求（24.9%关注） | 规模化部署的核心基础设施 |
| **自主改进** | 萌芽期（Engine 等产品出现） | 从人工循环到 autonomous loop |

**LangSmith Engine**（自主改进循环）在 Interrupt 2026 上发布，正好填补了「评估→改进」闭环的空白。
