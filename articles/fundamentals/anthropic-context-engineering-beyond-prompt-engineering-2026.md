# Context Engineering：超越 Prompt Engineering 的注意力管理科学

> 当 Prompt Engineering 解决的是「怎么写」，Context Engineering 解决的是「放什么进上下文」。2025 年 9 月，Anthropic 正式提出这个区分，揭示了一个更本质的工程问题：如何在有限注意力预算下，让模型始终维持在正确方向上。

---

## 一、从 Prompt 到 Context：工程重心的转移

Prompt Engineering 的核心问题是「措辞」：用什么词、什么顺序、什么例子能让模型输出更符合预期。这是 2022-2024 年 AI 应用层的主要工作内容。

但当模型被用于**多轮、长时序的 Agent 任务**时，问题发生了质变：

> "Context engineering is the art and science of curating what will go into the limited context window from that constantly evolving universe of possible information."
>
> （Context Engineering 是从不断演变的信息宇宙中， curation 什么将进入有限上下文窗口的艺术与科学。）

此时的核心问题不再是「怎么写」，而是「放什么进去」——因为上下文窗口是有限的，而 Agent 在每个推理周期生成的信息量远大于它能消费的。

---

## 二、注意力稀缺是架构性约束，而非阶段性问题

LLM 基于 Transformer 架构，每个 token 可以 attend 到上下文中的所有其他 token。对于 n 个 token 的上下文，存在 n² 个成对关系。

这导致了一个**天然的张力**：上下文越长，模型维持精确注意力的能力越薄。Anthropic 将此称为「context rot」——上下文腐烂。

这个问题的根源不是模型能力不足，而是：

**1. 架构性约束**。Transformer 的注意力计算量随上下文长度呈二次增长，任何模型在足够长的上下文中都会出现注意力衰减。

**2. 训练数据分布偏差**。模型在训练时，短序列的分布远比长序列常见，因此模型对短上下文的处理更精确，对长上下文的依赖关系处理更弱。

**3. 位置编码插值的代价**。Extensions 如 RoPE 通过插值让模型处理更长上下文，但会牺牲 token 位置理解的精确度。

这意味着「更大上下文窗口」不等于「更好的上下文利用」。Context Engineering 的目标是在注意力稀缺的前提下，最大化有效信号密度。

---

## 三、Context 的组成解剖：系统性的工程化

Anthropic 将 Agent 可用的上下文分解为多个组成部分：

| 组件 | 来源 | 工程挑战 |
|------|------|---------|
| System Prompt | 开发者写入 | 精确性与灵活性的平衡 |
| Tools / MCP | 外部定义 | 按需引入而非全部暴露 |
| Message History | 多轮对话积累 | 何时截断、如何压缩 |
| External Data | RAG / 知识库 | 相关性检索 vs 上下文污染 |
| Agent 生成的内容 | 上一轮推理结果 | 选择性保留哪些信息 |

每个组件都有自己的工程维度，不能用同一套策略处理。

## 四、System Prompt 的「海拔」设计

System Prompt 是开发者最能直接控制的上下文组成部分。Anthropic 指出了一个关键设计原则——「海拔」（altitude）概念：

**过于复杂/具体的问题**：工程师将复杂的 if-else 逻辑硬编码到 system prompt 中，试图引导 Agent 的精确行为。这会导致脆弱性——模型对指令措辞的微小变化过于敏感，维护成本随时间急剧增加。

**过于抽象/通用的问题**：提供高层次指导，假设模型有「共享上下文」。结果是模型的行为缺乏具体信号，无法可靠地在边缘场景中给出预期输出。

**正确的海拔**：具体到足以给模型清晰的行为信号，同时足够抽象以提供强 heuristics 处理未见过的场景。

Anthropic 给出的建议是**分层组织 prompt**：

1. **角色层**：定义 Agent 的身份和职责边界
2. **原则层**：高层决策准则（非具体规则）
3. **示例层**：用少量高质量的（input → output）对来传递模式
4. **约束层**：明确的边界条件和绝对禁区

## 五、Context Engineering 的核心策略

### 5.1 选择性注入（Selective Injection）

不是所有可用的信息都应该被注入上下文。Context Engineering 的第一个原则是**最小化原则**：用最小的信号集达到目标效果。

评估标准：这条信息是否直接改变模型在此任务上的输出？如果不是，考虑排除。

### 5.2 时序性上下文窗口

Agent 运行时，上下文窗口的内容是动态变化的。不同阶段需要不同的上下文配置：

- **启动阶段**：需要详细的初始化上下文（项目结构、Agent 角色、当前状态）
- **执行阶段**：需要最小化上下文，专注于当前任务相关的信息
- **交接阶段**：需要结构化的总结上下文，为下一阶段提供足够的上下文但不冗余

### 5.3 信息压缩与摘要

对于多轮对话历史，简单的截断（tail truncation）会丢失重要信息。更好的策略：

**语义摘要**：不是截断最后 N 条消息，而是对整个历史进行语义压缩，保留核心事件和决策点。

**增量状态记录**：不在消息历史中记录状态，而是用专门的状态字段维护当前进度（如 progress file），Agent 在每个循环中更新，读取时只读这一份结构化记录。

### 5.4 工具的按需加载

MCP 等协议允许动态引入工具定义。但将所有可用工具塞进上下文是反模式的——工具定义本身消耗上下文 token，且会稀释模型对真正相关工具的注意力。

**按需引入原则**：只在当前任务需要的时刻引入对应的工具定义，而不是预先暴露所有工具。

## 六、Context as Code：工程化的实践框架

Anthropic 提出的 Context Engineering 实际上是一个**工程化实践框架**，而非单个技巧。它的核心思想可以归纳为：

**1. 上下文是设计出来的，不是堆出来的。**

传统的做法是「把所有可能相关的信息都塞进上下文」。Context Engineering 的做法是「先确定目标行为，再设计最小的必要上下文」。

**2. 上下文随时间演变，需要持续 curation。**

不同于静态的 prompt 编写，Context Engineering 是一个**迭代过程**——在 Agent 的每个推理周期，决定保留、删除、压缩哪些上下文内容。

**3. 测试的是上下文质量，不是上下文数量。**

最终的验证标准是 Agent 在目标任务上的表现，而不是上下文中堆了多少信息。

## 七、与传统软件工程的类比

| 维度 | 传统软件工程 | Context Engineering |
|------|------------|-------------------|
| **资源** | CPU / 内存 | 注意力预算 |
| **优化目标** | 性能 / 内存占用 | 信号密度 / 注意力效率 |
| **调试方式** | profiler + log | 上下文窗口采样分析 |
| **设计原则** | 模块化 / 最小化依赖 | 最小化上下文 / 选择性注入 |
| **质量验证** | 单元测试覆盖 | 目标行为测试套件 |
| **最佳实践** | YAGNI / KISS | 最小上下文原则 |

## 八、实践建议清单

以下是从 Anthropic 文章中提炼的 Context Engineering 实践建议：

- [ ] **分层 System Prompt**：角色 → 原则 → 示例 → 约束，避免大段 if-else 硬编码
- [ ] **最小化上下文注入**：在每个注入决策前问「这条信息是否改变模型输出」
- [ ] **进度文件替代消息历史**：用结构化状态文件而非截断式历史记录
- [ ] **按需工具加载**：只在当前任务需要时引入工具定义，而非全部暴露
- [ ] **机器可读日志格式**：ERROR/INFO 格式而非人类友好的长篇日志
- [ ] **增量采样策略**：大测试套件用随机采样而非全量运行，避免上下文污染
- [ ] **语义压缩**：对多轮历史进行摘要而非简单截断
