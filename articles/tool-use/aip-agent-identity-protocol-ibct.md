# AIP：面向 MCP 和 A2A 的可验证代理身份协议

> **本质**：MCP 和 A2A 均不验证 Agent 身份——AIP 通过 IBCT 令牌为跨协议代理调用提供密码学级别的身份、授权和溯源机制

## 一、基本概念

### 问题：缺失的 Agent 身份层

当前 AI Agent 主要通过两个协议与外部世界交互：

- **MCP（Model Context Protocol）**：Agent 调用工具的协议
- **A2A（Agent-to-Agent）**：Agent 之间相互委托任务的协议

然而，**两个协议均不验证调用者的身份**。

Prakash (2026) 的扫描研究覆盖了约 2000 个 MCP 服务器，发现**全部缺乏身份认证机制**。这意味着：

- 任何恶意 Agent 都可以伪装成受信任的工具服务器
- A2A 委托链路上的下游 Agent 无法验证上游 Agent 的真实身份
- MCP 工具调用的授权边界无法跨协议追踪

### 解决方案：IBCTs

**Invocation-Bound Capability Tokens（IBCTs）** 是一种将身份、衰减式授权和溯源绑定融合进单一不可伪造令牌链的密码学原语。

核心设计目标：
1. **公钥可验证委托**：调用者身份可通过公钥密码学验证
2. **持有者侧衰减**：权限在传递过程中逐层递减，防止过度授权
3. **链式策略**：支持多跳委托的 Datalog 策略表达
4. **跨协议绑定**：同时支持 MCP / A2A / HTTP 传输层
5. **溯源完成记录**：每次调用产生可审计的完成证明

## 二、核心技术机制

### 两种 IBCT 模式

#### 1. 紧凑模式（Compact Mode）

适用于**单跳场景**：

- 格式：**签名的 JWT**
- 验证延迟：**0.049ms（Rust）/ 0.189ms（Python）**
- 相比无认证的增量开销：**0.22ms**
- 适用场景：MCP 工具调用、受信任 A2A 委托

#### 2. 链式模式（Chained Mode）

适用于**多跳委托链**：

- 格式：**Biscuit Token + Datalog 策略**
- 核心能力：Datalog 可以表达复杂的委托策略（如「只允许调用这 3 个工具」「调用深度不超过 5 层」）
- 支持权限衰减：每经过一个中间 Agent，权限自动递减

### IBCT 令牌链结构

```
[原始调用者身份]
    ↓ 创建 IBCT
[IBCT v1: 原始权限集]
    ↓ 委托给 Agent B（衰减）
[IBCT v2: 缩减权限集]
    ↓ 委托给 Agent C（再次衰减）
[IBCT v3: 最小权限集]
    ↓ 最终执行
[完成记录：可溯源的操作证明]
```

关键属性：**单向不可逆**——每次委托只能衰减，不能扩展权限。

## 三、安全评估

### 对抗性测试结果

研究团队在两个协议上各模拟了 **300 次攻击**（共 600 次），结果：

- **100% 拒绝率**：所有攻击尝试均被 IBCT 检测并拒绝
- 两种攻击类型**仅在 IBCT 链式模型下可检测**，在 unsigned 或纯 JWT 方案中完全无法发现：

#### 攻击类型 1：委托深度违反（Delegation Depth Violation）

攻击者试图突破委托层数限制，伪造更深层的委托链：

- unsigned 方案：无法检测，可任意深度伪造
- 纯 JWT：无法检测，因为 JWT 本身不携带深度信息
- IBCT 链式：每层委托在 Biscuit 令牌中明确记录深度，违规即拒

#### 攻击类型 2：空上下文审计规避（Audit Evasion Through Empty Context）

攻击者在多跳委托中通过传递空上下文来抹除操作记录：

- IBCT 链式：在每跳记录中强制包含「前驱上下文哈希」，空上下文会导致哈希不连续，被立即识别

### 何时考虑引入 AIP

如果你的 Agent 系统满足以下任一条件，应评估 AIP：

## 六、局限性

1. **证书管理**：IBCT 的安全性依赖公钥基础设施（PKI），部署者需要管理密钥生命周期
2. **Biscuit 生态成熟度**：相比 JWT，Biscuit 生态较新，生产工具链支持有限
3. **协议改动要求**：需要在 MCP 和 A2A 协议中增加 IBCT 支持，不是向后兼容的增量
4. **信任根问题**：IBCT 可以验证调用者身份，但初始信任根的建立（如「谁首次颁发证书」）仍需外部机制
