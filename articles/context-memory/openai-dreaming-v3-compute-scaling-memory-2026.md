# Dreaming V3：OpenAI 如何将记忆系统的工程代价压缩 5 倍

> 本文基于 OpenAI 官方博客 "Dreaming: Better memory for a more helpful ChatGPT" (2026-06-04) 深度分析

## 核心命题

**记忆系统的工程挑战不只是「怎么记」，更是「怎么负担得起地记」。OpenAI Dreaming V3 最大的工程突破不是功能演进，而是将后台记忆合成的计算成本压缩 5 倍——这个数字才是让免费用户也能用上动态记忆的关键。**

---

## 一、问题的本质：记忆的「过期腐烂」与「计算负债」

当我们讨论 AI Agent 的记忆系统时，最大痛点通常被描述为「记不住」或「记错了」。但 OpenAI 提出的框架揭示了更根本的问题：

1. **记忆会过期**：你说「我七月要去新加坡」，一年后 AI 还在推荐新加坡的酒店，因为它不知道时间已经过去了
2. **记忆需要主动修正**：Saved Memories 的设计是被动的——你说它记什么；但「我最近开始吃素了」这类隐含偏好变化，它完全感知不到
3. **记忆规模化的计算成本**：当用户量从百万级扩展到数亿级，时间跨度从几个月扩展到几年，后台合成记忆的计算代价成了商业化的核心障碍

> 引用原文：  
> *"Today we’re beginning to roll out a more capable and scalable system for synthesizing memory, developed to tackle the staleness, correctness, and scalability challenges that we observe when memory is applied to the hundreds of millions of users and multi-year time horizons in ChatGPT."*

第三点——**scalability**——才是 V3 真正解决的问题。功能上 V3 相比 V0 有进步，但真正让 V3 可落地的，是 5x 的计算效率提升。

## 二、三代记忆系统对比

### 2.1 演进路径

| 版本 | 时间 | 核心机制 | 局限 |
|------|------|---------|------|
| **Saved Memories** | 2024年4月 | 用户显式触发存储 | 依赖「请记住...」指令；自然对话内容被忽略；时间维度失效 |
| **Dreaming V0** | 2025年4月 | 后台自动从对话历史合成记忆 | 不能作为独立记忆系统；计算成本高，难以规模化 |
| **Dreaming V3** | 2026年6月 | 计算效率 5x 压缩；可独立作为记忆系统；用户可审查 | — |

V0 相比 Saved Memories 是质变——从被动存储到主动合成。V3 相比 V0 是量变+工程优化——记忆质量提升的同时，计算成本降到了可服务免费用户的水平。

### 2.2 三个可量化的记忆目标

OpenAI 提出了一套可评测的记忆质量框架，而不是「感觉更好」这种主观描述：

**① 携带上下文（Carry forward context）**
用户告诉 ChatGPT 一次的事情，后续对话能自动利用。eval 方式是构造需要回忆用户事实的问题，正确利用上下文得 1 分，否则得 0 分。

**② 遵循偏好（Follow preferences）**
偏好有三种形态：显式指令（「不要提 Stan」）、个人约束（「我是素食者」）、隐含偏好（「我住旧金山」→ 本地选项要据此调整）。eval 方式是检验模型是否正确应用记忆中的偏好。

**③ 随时间更新（Stay current over time）**
这是最有工程挑战的目标。「用户正在计划下周六的生日派对」这个记忆，必须在周日到来时自动更新为「用户的生日派对已经结束」。Dreaming 的后台机制使得这种时间敏感的更新成为可能。

> 引用原文：  
> *"With dreaming, memories are automatically updated as time passes, allowing ChatGPT to revise its memory from 'You're going to Singapore in July' to 'You went to Singapore in July 2026' when the trip ends."*

## 三、V3 的核心工程突破：5x 计算压缩

这是本文最值得深入分析的点。

### 3.1 为什么计算效率是瓶颈

Dreaming 的后台合成机制需要在每次对话后触发，扫描对话历史、提取模式、更新记忆状态。当服务规模从 Plus/Pro 用户扩展到免费用户（数亿级别），这个后台计算成本成了商业化门槛。

传统的做法是直接优化模型推理效率。但 V3 的 5x 压缩很可能来自几个工程维度：

- **记忆合成的触发优化**：不是每次对话都触发完整合成，而是基于信号强度（对话长度、主题相关性、偏好相关性强弱）动态决定是否触发
- **增量更新而非全量重建**：V0 可能是每次都重建完整记忆状态；V3 改为增量更新，只修正变化的部分
- **分层记忆架构**：高频变化的短期记忆和低频变化的长期记忆分层处理，避免每次都触发全局合成

### 3.2 5x 压缩的意义

OpenAI 原文说：

> *"Recent improvements reduced the compute required to serve dreaming to Free users by approximately 5x, making it possible to begin rolling out dreaming to Free users over the coming weeks and to increase memory capacity for Plus and Pro users."*

5x 这个数字的战略意义是：它把「dreaming 作为 Plus/Pro 专属功能」变成了「dreaming 可以服务数亿免费用户」。这意味着：

1. **记忆系统从付费墙功能变成基础设施**：当免费用户也能享受动态记忆，记忆就成了 ChatGPT 的默认能力而非增值功能
2. **用户数据的量级变化**：数亿免费用户的记忆数据使得 eval 数据更丰富，模型可以从更多真实交互中学习
3. **工程优先于算法**：OpenAI 在这里展示的工程思路是——当算法能力遇到商业化瓶颈，与其继续优化算法本身，不如在工程层面找到 5x 的效率空间

笔者认为，这个 5x 压缩的具体技术方案值得后续深入追踪。如果有内部技术报告或 ICML/ NeurIPS 论文，这会是理解 LLM 记忆系统工程极限的重要案例。

## 四、用户可审查的记忆摘要页

V3 还新增了一个产品层面的重要能力：**记忆摘要审查页**。

用户可以在 memory summary page 看到 ChatGPT 对自己的记忆摘要，快速了解「AI 知道我什么」，并可以主动添加、修正或删除信息，还可以给出指令（「关于 XX 话题，ChatGPT 应该在什么时候提起」）。

这解决了一个重要的信任问题：当用户不知道 AI 记住了什么，就无法信任 AI 的记忆。审查页让记忆从「黑箱」变成了「灰箱」——用户至少知道记住了什么，虽然合成过程仍然是 AI 内部的。

## 五、工程启示：记忆系统的三层架构

结合 Dreaming V3 和 CowAgent（Round349）的三层记忆架构，我们可以归纳出一个通用的 Agent 记忆系统设计模式：

```
┌─────────────────────────────────────────────────────┐
│              用户可感知层（审查/指令）                │
│  Memory Summary Page / 主动添加/修正/删除            │
└─────────────────────────────────────────────────────┘
                         ↑
┌─────────────────────────────────────────────────────┐
│           动态合成层（Dreaming / 主动更新）            │
│  后台进程：从对话历史提取模式 / 修正过时记忆            │
│  时间敏感更新 / 偏好归纳                              │
└─────────────────────────────────────────────────────┘
                         ↑
┌─────────────────────────────────────────────────────┐
│            静态存储层（Saved Memories）               │
│  用户显式触发 / 强信号存储                             │
└─────────────────────────────────────────────────────┘
```

三层各有分工：静态层处理强信号显式记忆；动态层处理弱信号隐式模式；感知层处理用户审查和指令。这个分层架构比「把所有记忆塞进一个向量数据库」要工程上更可持续。

## 六、与 Round350 Cursor Lessons 的关联

Round350 的 Cursor 云端 Agent 工程教训提到了「环境是模型能力的乘数」。这个观点在 Dreaming V3 中得到了另一个维度的印证：

记忆系统也是 Agent 能力的乘数。一个有动态记忆的 Agent 不需要每次从头解释背景，它可以在「已经知道用户是谁、用户偏好什么、用户当前项目是什么」的前提下直接工作。

**这两篇文章共同指向一个结论**：云端 Agent 的基础设施不只是 Durable Execution 和环境隔离，还包括让 Agent 持续学习（记忆）和持续判断（Auto-review）的机制。没有这两层，Agent 在长任务中就会变成「每次都从头认识你」。

## 七、哪些已知问题尚未解决

Dreaming V3 仍有局限，OpenAI 官方没有完全披露：

1. **记忆合成的延迟**：后台合成是异步的，这意味着新对话开始时记忆可能还没有更新完成。用户能否接受「刚说的偏好还没生效」这个时间差？
2. **隐私边界**：当数亿免费用户的记忆数据成为模型训练信号，隐私保护的具体机制是什么？
3. **记忆冲突解决**：如果用户的显式指令（「记住我不喝酒」）和隐含行为（连续三周讨论精酿啤酒）产生矛盾，Dreaming 如何处理？

这些问题值得在下轮继续追踪。
