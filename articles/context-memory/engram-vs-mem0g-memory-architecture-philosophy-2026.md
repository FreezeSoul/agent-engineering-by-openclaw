# Engram vs Mem0g：两种记忆架构的设计哲学对比

> **摘要**：DeepSeek Engram 和 Mem0g 代表了 AI Agent 记忆系统的两条根本不同的路线：前者将记忆编译进模型权重，后者将记忆作为外部可检索系统。本文从设计哲学、记忆模型、检索机制三个维度深度对比，揭示"把记忆放进模型"与"给系统接入记忆"的本质权衡。

## 一、为什么这两个系统值得对比

2026 年初，DeepSeek 发布 Engram 条件记忆模块，Mem0 发布 Mem0g 图增强记忆系统。两个系统都叫"记忆"，但它们的解决思路截然不同——甚至可以说代表了两个完全不同的技术哲学。

**Engram** 把记忆看作模型的一部分：它修改 Transformer 架构本身，通过 n-gram 哈希查找表在模型层内实现 O(1) 知识检索，目标是让模型"记住"高频模式从而释放注意力容量。

**Mem0g** 把记忆看作系统的外部资源：它维护一个独立的图结构记忆库，通过向量检索+图遍历的混合机制实现动态记忆读写，目标是让 Agent 在多轮对话中追踪实体关系和时序变化。

这两条路线不是竞争关系，而是分别回答了不同层次的问题。本文用三个维度系统对比它们的本质差异。

## 二、设计哲学：把记忆编译进模型，还是独立于模型？

### Engram：记忆是模型的隐性知识

Engram 的核心假设是：LLM 本身就是一种记忆系统，只是这种记忆以权重形式存储，无法高效读取。Engram 论文的出发点不是"给 LLM 增加记忆"，而是"让 LLM 已经记住的东西能够被快速访问"。

传统 Transformer 在处理高频静态模式（如"中国的首都是北京"）时，仍然要动用多层注意力来"推理"出答案。Engram 认为这是浪费——这类知识模型已经"记住"了，只是访问效率太低。解决思路是把这些静态模式从计算密集型的注意力计算中"卸载"到专门的哈希查找表中。

这种思路可以追溯到经典 NLP 中的 n-gram 平滑技术：N-gram 模型通过统计共现频率来估计概率，本质上是一种"记忆"。Engram 的创新在于把这种思想现代化，并用可学习的哈希查找表替代了传统查表，使记忆的读取变得端到端可微。

关键约束：**Engram 记忆是只读的静态知识**。它不能在线更新——除非重新训练模型或执行一次性的模型编辑（model editing）。这意味着 Engram 记忆处理的是"模型应该永久知道什么"，而非"Agent 当前会话中发生了什么"。

### Mem0g：记忆是独立演化的外部状态

Mem0g 的设计哲学截然相反：记忆不是模型的属性，而是独立于模型的外部状态。模型只负责推理，记忆由专门的记忆管理系统负责存储、检索和更新。

Mem0g 的记忆是**动态的**：每次 Agent 与环境交互后，结果可以写入记忆；新的上下文可以触发记忆的检索和引用；记忆中的错误信息可以在后续交互中被修正。这是典型的主体-记忆分离架构（decoupled memory）。

这种思路的优势是灵活性：记忆的更新不需要重新训练模型，不需要修改模型权重，不需要模型本身支持任何特殊机制。任何能输出文本的 LLM 都可以接入 Mem0g 记忆系统。

代价是：记忆的正确性、一致性和更新策略都成为需要额外设计的工程问题。模型本身无法"理解"记忆，只能把记忆作为上下文的一部分来处理。

### 本质差异总结

| 维度 | Engram | Mem0g |
|------|--------|-------|
| 记忆的拥有者 | 模型本身 | 外部系统 |
| 记忆的更新方式 | 训练时固化，只读 | 在线读写，可动态更新 |
| 记忆的内容类型 | 高频静态知识模式 | 实体关系 + 时序事件 + 会话上下文 |
| 记忆的访问延迟 | O(1)，可预取到 CPU RAM | 取决于向量检索+图遍历，通常数十到数百毫秒 |
| 对模型架构的要求 | 需要修改模型权重/架构 | 零模型侵入，任何 LLM 均可接入 |
| 适用场景 | 基础模型层面的知识密度增强 | Agent 会话层面的上下文管理 |

## 三、记忆模型：哈希查表 vs 图结构

### Engram 的记忆模型：哈希 n-gram 嵌入表

Engram 的记忆存储是一个**哈希查找表**，key 是 token 序列的 n-gram 哈希值，value 是对应的"记忆激活"向量。这个表在训练阶段构建：将大量文本语料中的 n-gram 统计模式提取为稀疏向量，存入外部 DRAM；推理时，给定当前 token 序列，计算其 n-gram 哈希，直接查表获取记忆激活。

Engram 论文中给出 Engram-27B 的具体配置：
- Engram dimension (`d_mem`): 1280
- Engram vocab size: 2,262,400（27B 模型）
- N-gram 范围: [2, 3]（二元和三元）
- 插入位置: Transformer 层 [2, 15]（不是全部层，只在特定层注入记忆）

Engram 的稀疏性体现在：不是所有 token 都有对应的记忆条目——只有高频出现的静态模式（stereotyped patterns）才会被编码进 Engram 表。那些低频但模型仍然需要处理的信息，仍然由正常的注意力机制处理。

关键洞察：Engram 解决了"模型已经知道但无法快速访问"的问题，而不是"模型还不知道"的问题。如果一个模式在训练语料中从未出现，Engram 无法凭空创造记忆。

### Mem0g 的记忆模型：混合向量+图结构

Mem0g 在 Mem0 基础上增加了**图增强层**。其记忆库由两个核心组件构成：

**向量存储层**：所有记忆条目首先被编码为向量，存储在向量数据库中。这一层负责语义相似性检索——给定当前查询，找到语义最相关的记忆条目。检索是 O(log n) 的近似最近邻（ANN）查询。

**图结构层**：在向量检索之上，Mem0g 额外维护一个实体关系图。图的节点是实体（人物、地点、事件），边是实体之间的关系类型和时序顺序。图结构使得多跳推理成为可能——通过图的遍历，可以找到"某人在某事件之前做的另一件事"这类时序关系。

Mem0g 在 LOCOMO benchmark 上取得 58.13% 的时序推理准确率，领先其他系统 36 个百分点。这说明图结构确实在需要关系推理的场景中发挥了作用。

Mem0g 的记忆内容是**由 Agent 运行时生成的**——每次 Agent 执行工具调用、获得反馈、做出决策，相关信息都会被编码后存入向量库和图结构。这是一种 append-only 的记忆积累策略（不涉及显式遗忘机制）。

### 对比分析

Engram 的哈希表是**压缩的、只读的、统计驱动的**——记忆内容由训练数据决定，反映的是语言中反复出现的模式。

Mem0g 的图+向量是**宽松的、可写的、由 Agent 经验驱动的**——记忆内容由 Agent 的具体交互决定，反映的是这个特定 Agent 在特定环境中的经历。

两者的关系更像是互补而非竞争：Engram 可以视为模型内部知识库的加速器，Mem0g 可以视为 Agent 的外部经验日志。一个 Agent 可以同时使用两者——用 Engram 处理通用知识，用 Mem0g 处理任务特定的上下文。

## 四、检索机制：O(1) 确定性查找 vs 混合遍历

### Engram 的检索：确定性 O(1)，可硬件级预取

Engram 的检索过程是**确定性的**：
1. 给定输入 token 序列 `[t_1, t_2, ..., t_n]`
2. 提取所有 n-gram：`[t_1 t_2], [t_2 t_3], ..., [t_{n-1} t_n]`（bigrams）和 `[t_1 t_2 t_3], ...`（trigrams）
3. 对每个 n-gram 计算哈希值：`h = hash(n_gram)`
4. 查表获取 `Engram[h] = v`（d_mem 维向量）
5. 将记忆激活向量注入对应 Transformer 层的残差流

关键优势：**确定性寻址**意味着内存访问模式在输入确定时完全已知。这使得硬件预取成为可能——在模型处理当前 token 时，下一层的 Engram 查找已经可以开始预取。Engram 论文测量显示，即使 Engram 表存储在 CPU DRAM 中，HBM 带宽限制的缓解带来的收益也远超内存访问延迟的代价。

### Mem0g 的检索：混合向量召回 + 图扩展 + 重排序

Mem0g 的检索是**多阶段的**：

```
输入查询 Q
  ↓
[阶段1] 向量语义召回
  从向量库 ANN 检索 Top-K 相关记忆
  ↓
[阶段2] 图结构扩展
  对召回的记忆实体，在图中找其一跳邻居实体
  应用时序过滤：保留发生在当前上下文时间窗口内的事件
  ↓
[阶段3] 重排序（Rerank）
  用交叉编码器对图扩展后的记忆与 Q 做相关性重排
  ↓
[阶段4] Top-N 输出
  将重排后的记忆注入 Agent 上下文
```

LOCOMO 基准测试中，Mem0g 的图扩展阶段贡献了主要的时序推理能力提升——向量召回单独无法区分"A 在 B 之前发生"这类时序关系，加入图遍历后才能正确处理。

Mem0g 的检索延迟远高于 Engram（毫秒级 vs 微秒级），但它的表达能力也强得多：可以基于语义相似性、实体关系、时序约束等多个维度检索，而 Engram 只能基于 token 共现模式。

## 五、工程视角：什么时候用哪个？

### 选择 Engram 的场景

**场景1：想提升模型的基础知识密度，但无法修改模型权重**
Engram 作为模型层面的架构修改，普通开发者无法直接使用。但 DeepSeek 已经发布了集成 Engram 的模型（DeepSeek-V4-Engram 等）。如果你使用 DeepSeek 系列模型，已经自动获得了 Engram 记忆能力。

**场景2：推理效率是核心瓶颈，且知识以静态模式为主**
Engram 的确定性 O(1) 查找使得记忆访问几乎不产生额外延迟。如果你的应用场景高度重复（如客服机器人处理同类问题），Engram 可以显著降低平均推理成本。

**场景3：需要在资源受限环境下运行大模型**
Engram 通过把记忆表卸载到 CPU DRAM，理论上可以让模型在更小的 HBM 占用下保持同等知识密度。这对端侧部署有一定意义。

### 选择 Mem0g 的场景

**场景1：Agent 需要追踪多轮交互中的实体状态变化**
这是 Mem0g 最典型的应用场景。Agent 在执行任务过程中不断累积关于环境状态的信息（文件内容、API 返回、用户反馈），这些信息需要被记忆并在后续决策中被引用。

**场景2：需要在记忆中进行多跳关系推理**
当 Agent 需要回答"这个项目上次失败是因为什么，后来做了什么修改？"这类问题时，图结构的 Mem0g 明显优于纯向量检索。

**场景3：希望记忆系统独立于特定模型**
Mem0g 对模型完全无侵入。如果你的 Agent 需要在多个模型之间切换（或者同时使用多个模型），外部记忆系统是最灵活的方案。

### 不能互相替代的根本原因

Engram 和 Mem0g 解决的是不同层次的问题：

- **Engram** 回答"模型应该如何高效地储存和读取它已经学会的知识"
- **Mem0g** 回答"Agent 在运行时应该如何积累和检索它尚未学会的、来自具体经验的知识"

类比人类认知：Engram 像人的长期记忆（语义记忆），存储在神经网络权重中，通过练习固化；Mem0g 像人的情景记忆（episodic memory），记录具体经历，可以随时读取和更新。两者功能互补，共同构成完整的记忆系统。

## 六、判断与局限

### 两条路线都有各自的盲点

**Engram 的盲点**：只读记忆意味着它无法适应快速变化的世界。如果"某某公司刚发布了新产品"这类实时信息，Engram 无能为力——它需要重新训练或执行模型编辑才能更新。此外，Engram 的 n-gram 哈希方法本质上只能捕获 token 级别的共现模式，对于更高层次的语义关系（图谱结构、因果链条）无能为力。

**Mem0g 的盲点**：图结构的维护成本高。实体消歧、关系抽取的准确性直接影响图的质量；如果向量召回阶段检索到了不相关的记忆，会污染后续的推理过程。Mem0g 的记忆检索本身也是一个 LLM 调用（用于判断什么值得记忆、如何编码），这引入了额外的延迟和不确定性。

### 一个尚未验证的假设

笔者认为，2026 年最有价值的 Agent 记忆架构创新，可能不是 Engram 或 Mem0g 单独演进，而是**两者的结合**：用 Engram 模式处理通用语义知识（高效、O(1)），用 Mem0g 模式处理任务特定经验（灵活、可检索）。Agent 的记忆系统可以设计为两层：

```
Agent 记忆系统架构（假设）
├── Engram 层（模型内）
│   └── 处理通用高频知识模式（O(1) 查找，零额外延迟）
└── Mem0g 层（模型外）
    ├── 向量层：任务特定的语义上下文
    └── 图层：实体关系和时序追踪
```

这种架构目前没有公开实现，但它在逻辑上是自洽的——每层记忆处理它最擅长处理的信息类型。
