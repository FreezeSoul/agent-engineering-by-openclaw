# CrewAI：你的第一个 AI Agent 应该只做一件事，而且做得不好

> **核心命题**：Agent 系统不是设计出来的，是迭代出来的。最有效的落地路径是从一个最小化、有缺陷的版本开始，通过真实运行数据逐步进化，而不是预先设计一个「完美架构」。
>
> 来源：[Your First AI Agent Should Do One Thing Badly](https://www.crewai.com/blog/your-first-ai-agent-should-do-one-thing-badly)（CrewAI Blog）

---

## 从 2 Billion 工作流中学到的教训

CrewAI 过去一年观察了数十个企业部署、数以十亿计的工作流执行，发现了一个清晰的模式：

**团队普遍卡在「demo 演示」和「生产系统」之间的沟壑里。**

PPT 里的架构看起来很漂亮，但真实用户行为会撕碎所有预设假设。真正能交付价值的团队，并不是技术更强，而是方法论更务实。

---

## 迭代优先于设计

CrewAI 观察到最有效的 agent 团队，都遵循一个反直觉的原则：

> **从「做得不好」开始，而不是从「设计完美」开始。**

这与传统软件工程的教育完全相反。我们被训练去预判边缘情况、从第一天就设计可扩展性。但 agent 系统不同——它的行为是从 prompts、tools、execution context 的交互中**涌现**出来的，无法在设计阶段完全预知。

一个典型场景：

> 一支由资深工程师组成的团队，设计了一个包含 researcher agent → planner agent → executor agent → quality checker 的完整多 agent 架构。架构文档完善、README 出色。
>
> 三个月后，项目还在开发中。零生产流量。

这个故事的关键不是「 ambition 太强」，而是「在还不理解系统的时候就优化了一个还不存在的系统」。

## 最小化启动的实际案例

**医疗人员认证场景**：一个医疗机构需要处理医生的执照验证、医学会查询、制裁名单检查、身份验证。手动流程是个巨大瓶颈。

他们的选择：**只自动化背景调查这一个工作流。**

初始版本：
1. 收集 provider 数据
2. 查询相关来源
3. 结构化输出 JSON
4. 基于数据做决策
5. 推送到 Snowflake

没有覆盖所有边缘情况。几周就交付了。

真实运行后，他们逐步加入了：
- hallucination 检测的 guardrails
- 合规审计跟踪
- 并行工作流处理

最终流程从**天级别缩短到小时级别**。第二个用例已经在开发中。

如果一开始就设计完整的系统，这个时间表是不可能的。

## Human-in-the-Loop 是特性，不是限制

CrewAI 观察到的另一个关键洞察：

**信任是通过生产运行挣来的，不是设计出来的。**

最有效的团队不是从「full autonomy」开始的，而是从「100% 人工审查」开始，然后基于真实执行数据逐步降低人工介入比例。

一个 HR 服务公司的案例：
- 3000+ 员工工单/月
- 合规团队疲于应对
- Agent（名叫 Andy）起草回复、呈现相关政策、标记边缘情况
- **每一张票都经过人工审查后才到达员工**

然后有趣的现象发生了：合规团队在几千次执行后注意到质量足够高，SVP 问：「我们能用这些输出来训练我们的人类 agent 吗？」

这意味着他们可以开始降低监管比例了。现在 Andy 处理 **50%+ 的接触无需人工审查**，因为 agent 通过数千次一致执行赢得了信任。

## 失败应该是显而易见的

传统软件工程倾向于「优雅降级」——出错了不要完全崩溃，保持服务降级运行。

但对于 agent 系统，**这个策略是反效果的**。

在测试阶段，应该让失败变得明显、夸张、难以忽视——而不是优雅地降级。你需要清楚地看到失败，才能理解失败。

到了生产阶段，再考虑优雅降级。

---

## 当你有多 agent 设计冲动时，先问自己这个问题

> "Add agents when you have evidence, not intuition."

「我觉得我们需要一个 validator」是一个猜测，不是判断。

更好的问题是：「47% 的错误是格式问题，一个 validator 会抓到吗？」——这是数据，不是直觉。

多 agent 架构可以很强大，但它们也成倍增加了调试复杂度。在有真实数据证明需要细分之前，单 agent 或者简单 pipeline 通常是正确答案。

## 闭环关联

| 层次 | 组件 | 作用 |
|------|------|------|
| **设计哲学（本文）** | 迭代优先于设计 | 从最小可行版本开始，通过数据进化 |
| **执行机制（关联项目）** | microVM / Agent spawning | 快速复制 agent 实例做并行实验 |
| **评估体系（关联工具）** | Agent eval harness | 判断哪个版本真正改进了 |

CrewAI 的洞察与 LangChain 的 [State of Agent Engineering 2026 报告](https://www.langchain.com/state-of-agent-engineering)（1340 人调研）形成呼应：57% 的团队已有生产 agent，但信任和评估仍然是最大障碍。本文提供了「信任如何通过迭代Earn」的实操路径。

---

## 结论

> 不要构建让你印象深刻的东西，构建你能够快速交付并从中学习的东西。
>
> 如果一个版本花费超过两周，你可能构建得太多了。

成本足够低，迭代速度足够快，不需要在一开始就做到完美。
