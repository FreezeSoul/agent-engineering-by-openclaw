# CAMP：案例自适应的多智能体评议框架

> **本质**：动态组建专科医生面板 + 三值投票 + 层级仲裁——解决"一个面板不适用所有病例"的多智能体临床预测难题

## 一、基本概念

### 1.1 问题：案例级异质性

LLM 应用于临床预测任务时，输出对 prompt 框架、专科视角和推理路径高度敏感。但这种不稳定性并非均匀分布：

- **简单病例**：无论配置如何变化，输出保持一致
- **复杂病例**（重叠病症或证据模糊）：推理配置的小幅变化就会导致输出分歧

这被称为**案例级异质性（Case-level Heterogeneity）**，揭示了静态推理策略与个体病例异构推理需求之间的结构性错配。

### 1.2 现有方法的局限

| 方法 | 局限 |
|------|------|
| CoT / Self-consistency | 从单一角色条件分布采样，多样性只是采样噪声，无法按病例调整专家构成 |
| 固定角色多智能体 | 所有病例使用相同角色集 + 扁平多数投票，丢弃了分歧中携带的诊断信号 |

### 1.3 CAMP 的核心思想

CAMP（Case-Adaptive Multi-agent Panel）模仿真实临床会诊过程：

1. **主诊医生**（Attending Physician Agent）：提取结构化证据，生成候选诊断，**动态组建专科面板**
2. **专科医生**（Specialist Agents）：三值投票（KEEP / REFUSE / NEUTRAL），在专业范围外**原则性弃权**
3. **混合路由**（Hybrid Router）：强共识 → 接受；无共识 → 仲裁（**权衡论证质量而非票数**）

> *不是"一个面板打天下"，而是"每个病例组建专属面板"*

## 二、核心技术机制

### 2.1 三阶段工作流

```
┌─────────────────────────────────────────────────────────┐
│  阶段一：主诊医生（Attending Physician）                      │
│  · 提取结构化证据                                            │
│  · 生成候选诊断列表                                          │
│  · 根据病例诊断不确定性动态选择相关专科医生，组建专属面板           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  阶段二：专科医生面板（Specialist Panel）                    │
│  · 每个专科独立评估每个候选诊断                                 │
│  · 三值投票：KEEP（保留）/ REFUSE（拒绝）/ NEUTRAL（弃权）        │
│  · 弃权 = "超出专业范围" 的原则性问题，而非不确定                │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  阶段三：混合路由（Hybrid Router）                          │
│  · 强共识路径：多数专科 KEEP → 直接采纳                       │
│  · 回退路径：无共识 → 主诊医生判断兜底                         │
│  · 仲裁路径：分歧明显 → 主诊医生审视各专科推理，权衡论证质量而非票数  │
└─────────────────────────────────────────────────────────┘
```

### 2.2 三值投票机制（Three-Valued Voting）

传统多智能体投票：二值（同意/拒绝）→ 丢失弃权信号

CAMP 三值设计：
- **KEEP**：该专科**确认**此诊断在专业范围内且证据充分
- **REFUSE**：该专科**拒绝**此诊断，证据不支持或超出专业
- **NEUTRAL**：超出专业范围，**不参与仲裁权衡**

关键洞察：NEUTRAL 不是"不确定"，而是"这个议题不在我能力圈内"——这种原则性弃权保留了分歧中的诊断信号。

### 2.3 动态面板组建（Case-Adaptive Assembly）

固定面板 vs 动态面板的核心区别：

| 特性 | 固定面板（现有方法）| 动态面板（CAMP）|
|------|------------------|----------------|
| 角色选择 | 所有病例同一套 | 按病例诊断不确定性选择 |
| 专科构成 | 扁平叠加 | 层级组合 |
| 专家适配 | 无 | 根据病例涉及的器官系统、模糊证据区域动态调整 |

主诊医生根据病例的临床表象决定咨询哪些专科——这与真实临床决策过程完全一致。

### 2.4 混合路由决策

```
                    ┌──────────────────────┐
                    │  强共识检测           │
                    │  多数 KEEP？          │
                    └──┬───────────┬───────┘
                    是 │           │ 否
                       ↓           ↓
                  [直接采纳]    [进入仲裁流程]
                                  │
                    ┌─────────────┴─────────────┐
                    ↓                           ↓
            [分歧专科数量少]              [分歧专科数量多]
```

## 三、与其他多智能体方法的关系

### 3.1 CAMP vs 固定角色多智能体

| 维度 | 固定角色多智能体 | CAMP |
|------|----------------|------|
| 面板构成 | 所有病例相同 | 每病例动态组建 |
| 投票机制 | 扁平二值/多数票 | 分层三值投票 |
| 分歧处理 | 丢弃诊断信号 | 仲裁时权衡论证质量 |
| 透明性 | 黑盒输出 | 投票记录 + 仲裁追踪全程可审计 |

### 3.2 与现有 Multi-Agent 架构的演进关系

```
Stage 7 (Orchestration)
    │
    ├── Plan-Execute（计划-执行分离）
    ├── Handoffs（角色交接）
    └── CAMP 贡献：动态专科面板组建 ──────────────┐
                                                   ↓
Stage 9 (Multi-Agent)                              │
    ├── Agent Q-Mix（RL 拓扑选择）                  │
    ├── CausalPulse（神经符号多智能体）               │
    └── CAMP 贡献：案例自适应 + 三值投票 + 仲裁路由 ──→ 新的设计模式
```

### 3.3 CAMP 的通用化启示

虽然 CAMP 评估在临床预测领域，但其设计模式具有跨领域泛化价值：

- **需要异构专家输入**的场景（法律、金融、工程）
- **案例复杂度差异大**（简单案例不需要专科，复杂案例需要精准专科组合）
- **需要可审计决策**（监管/合规要求高）

## 四、实验评估

### 4.1 任务

| 任务 | 描述 | 数据来源 |
|------|------|---------|
| 诊断预测（Diagnosis Prediction）| 从临床笔记预测诊断 | MIMIC-IV |
| 住院课程生成（BHC）| 生成简短住院治疗路径 | MIMIC-IV |

### 4.2 基线对比

CAMP 在 **4 个 LLM backbone** 上**一致超越**强基线：

| 方法类型 | 代表方法 |
|---------|---------|
| 单智能体 | Direct prompting, CoT, Self-consistency |
| 固定角色多智能体 | 固定角色面板 + 多数投票 |
| 路由方法 | 动态路由单一专科 |

**关键发现**：
- CAMP 超越所有基线的同时，**消耗 token 少于大多数竞争多智能体方法**
- 四个 LLM backbone 均呈现一致性提升（说明 CAMP 的通用性）

### 4.3 可审计性

- **投票记录**：每个专科对每个候选诊断的 KEEP/REFUSE/NEUTRAL 完整记录
- **仲裁追踪**：主诊医生的最终判断理由全程可追溯

## 五、工程实践指南

### 5.1 适用场景判断

CAMP 适用于以下条件（同时满足）：
- [ ] 任务需要**多个异构专家**的判断输入
- [ ] 案例**复杂度差异大**（简单/复杂需要不同处理策略）
- [ ] 需要**可审计的决策过程**
- [ ] **token 成本**是重要约束（相比固定面板更高效）

### 5.2 三值投票的工程实现

```python
# 三值投票的枚举定义
class Vote(Enum):
    KEEP = "keep"      # 专业范围内，证据充分
    REFUSE = "refuse"  # 专业范围外或证据不支持
    NEUTRAL = "neutral"  # 超专业范围，不参与仲裁

# 弃权信号保留的关键：NEUTRAL 不参与质量权衡
def aggregate_votes(votes: List[Vote]) -> Decision:
    keep_count = sum(1 for v in votes if v == Vote.KEEP)
    refuse_count = sum(1 for v in votes if v == Vote.REFUSE)
    neutral_count = sum(1 for v in votes if v == Vote.NEUTRAL)
    
    if keep_count > refuse_count + neutral_count:
        return Decision.ADOPT  # 强共识
    elif refuse_count > keep_count + neutral_count:
        return Decision.REJECT  # 强拒绝
    else:
        return Decision.ARBITER_ROUTING  # 进入仲裁
```

### 5.3 案例自适应面板的组装策略

```python
def assemble_panel(case: ClinicalContext) -> List[Specialist]:
    """根据病例诊断不确定性动态组建面板"""
    uncertainty = assess_diagnostic_uncertainty(case)
    
    if uncertainty.low:
        return [Generalist()]  # 简单病例：一个通科医生即可
    elif uncertainty.medium:
        return [Cardiologist(), Pulmonologist()]  # 中等：相关专科组合
    else:
        return [
            Cardiologist(), Pulmonologist(), 
            Nephrologist(), Neurologist()
        ]  # 高不确定性：更宽的专科覆盖
```

## 六、局限性与未来方向

### 6.1 当前局限

1. **专科池固定**：面板组建从预定义专科池选择，专科池本身的完备性影响效果
2. **MIMIC-IV 单领域**：目前评估在临床预测，跨领域泛化能力有待验证
3. **主诊医生 LLM 的仲裁质量**：仲裁质量高度依赖主诊医生 LLM 的推理能力

### 6.2 未来扩展方向

- **开放域 CAMP**：将临床专科池扩展为通用领域专家池
- **学习面板组建**：用 RL 或搜索方法学习最优面板组建策略
- **交叉专科投票**：同一专科可对多个候选诊断投票，多专科可参与同一诊断的评估
