# 贡献指南

> 为 Agent 工程知识体系贡献高质量内容

---

## 核心原则

**质量 > 数量**。宁可少发一篇，也不发低质内容。

---

## 收录标准

每一篇文章必须满足以下全部条件：

### 1. 解决实际问题 or 澄清认知误区

- 解决一个真实的 Agent 工程问题（架构设计、安全、评测等）
- 或澄清一个常见的认知误区
- **不收**：纯资讯、新闻报道、产品发布

### 2. 有核心 insight

- 有原创性见解，不是翻译或搬运
- 经过内化，有自己的判断
- 能回答"这篇文章让我对某个问题的理解发生了改变"这个问题

### 3. 对工程师有实战价值

- 提供决策参考（选型判断、权衡分析）
- 或提供实战指导（代码示例、架构模式、工程实践）
- 纯理论、无实践价值的文章不收

### 4. 内容经过内化

- 用自己的语言重新组织，而非复制粘贴
- 有判断、有评价，不只是总结他人观点
- 注明来源，但不等同于原创

---

## 文章结构规范

每篇文章头部必须包含：

```markdown
# 文章标题

> 一句话核心总结，说明本文解决什么问题或提供什么价值

## 标签（可选）
- `tag1` / `tag2` / `tag3`

## 来源（可选）
- 原始论文 / 博客链接
- 评分：X/5（实用性 / 独特性 / 时效性）
```

---

## 分类指引

将文章放入最合适的目录：

| 目录 | 内容 |
|------|------|
| `articles/fundamentals/` | 基础概念、设计模式、工程思维、Prompt Engineering |
| `articles/context-memory/` | 记忆架构、上下文管理、RAG 融合 |
| `articles/tool-use/` | MCP 协议、工具调用、安全实践 |
| `articles/orchestration/` | 多 Agent 协作、协议栈（A2A/MCP/A2UI）、编排模式 |
| `articles/harness/` | Harness Engineering、安全约束、评测框架 |
| `articles/evaluation/` | 评测基准、可观测性、Benchmark 分析 |
| `articles/deep-dives/` | 单点深度分析、范式研究、源码解读 |

---

## 不收录的内容

- ❌ **资讯快讯**：CVE 披露、版本发布、价格调整
- ❌ **周报/月报**：时间敏感的资讯汇总
- ❌ **纯教程**：无独特见解的入门指南、翻译教程
- ❌ **搬运内容**：未经内化的翻译或复制粘贴
- ❌ **营销内容**：产品宣传、软文

---

## 更新 changelog

新增文章后，更新 `changelog/SUMMARY.md` 中的对应分类表格。

---

## 提交流程

1. Fork 本仓库
2. 创建新分支：`git checkout -b articles/your-article-name`
3. 添加文章到对应目录
4. 更新 `changelog/SUMMARY.md`
5. 提交 PR
